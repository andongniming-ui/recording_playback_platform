## 1. 先理解这个平台是做什么的

AREX Recorder 是一个“**录制、回放、对比**”平台。

它最核心的想法很简单：

- 先把真实业务请求录下来
- 再把同样的请求重新发一次
- 最后比较两边结果是不是一致

你可以把它理解成一个“自动化回放测试平台”：

- 平时用户在系统里点按钮、提交表单、发请求
- 平台把这些真实请求记录下来
- 之后你可以把这些请求拿去别的环境再跑一遍
- 比如拿生产录制的数据去测试环境、回归环境跑
- 如果结果不一样，平台会告诉你哪里不一样

这类平台的价值在于：

- 不用手工一个个写接口测试
- 不用自己回忆接口参数
- 能直接拿真实业务流量做测试
- 能发现环境差异、代码回归、配置错误等问题

## 2. 平台的工作原理

这个平台不是“只看页面”，它背后有一整条流程。

### 2.1 录制是怎么来的

平台支持的是 **Agent 录制**。

简单说就是：

1. 在目标 Java 应用里挂一个 AREX Agent
2. 用户照常访问业务系统
3. Agent 在 JVM 里把请求、响应、子调用记录下来
4. 这些记录先上传到 AREX 的存储服务
5. 平台再把这些数据同步回来展示

你可以理解成：

- 业务系统照常运行，不需要改很多代码
- Agent 像“录音笔”一样在系统内部记录发生了什么
- 平台负责展示、筛选、转成用例、发起回放

### 2.2 录制数据里有什么

一条录制数据通常包含这些内容：

- 请求方法，比如 `GET`、`POST`
- 请求地址
- 请求参数
- 请求体
- 响应结果
- 耗时
- 交易码、业务码、接口识别码
- 子调用信息
- 关联的录制会话

如果你的业务系统会访问数据库，或者调用 Redis、RPC 接口，这些也可能作为“子调用”一起被记录下来。

### 2.3 同步是什么意思

很多新手会误解“同步”。

这里的同步不是“同步到目标系统”，而是：

- 目标系统已经被 Agent 录了
- 数据先放到存储服务里
- 平台点击“同步”后，再把这些记录拉回来

所以平台里的“录制会话”，本质是：

1. 先点击“开始录制”，把会话切到录制中
2. 让目标系统真实跑请求
3. 再点击“停止录制”，平台去存储服务拉取录制数据
4. 收集完成后，你就在页面上看和操作这些录制

### 2.4 回放是怎么来的

回放就是：

1. 从录制数据里取出原始请求
2. 把这条请求重新发到目标环境
3. 取回新响应
4. 跟录制时的结果做比较

回放的目的不是为了“再录一遍”，而是为了检查：

- 这个接口在新环境里还能不能正常工作
- 返回结果是不是跟原来一样
- 有没有业务字段差异

### 2.5 对比是怎么来的

对比通常是：

- 同一批录制请求
- 同时发到两个环境
- 比较两边响应是不是一样

比如：

- 左边发到 PL2
- 右边发到 VT
- 平台比较两边返回

如果返回不一样，就会显示差异字段。

## 3. 你可以用它做哪些事

这个平台不是只能“看录制”。

你可以用它做的事很多，按实际使用场景分，主要有这些：

### 3.1 录制真实业务请求

- 把真实环境的业务流量记录下来
- 比如开户、转账、查询、登录、提交表单等
- 录制后的数据可以继续做测试资产

### 3.2 从录制生成测试用例

- 录制不是终点
- 录制数据可以直接转成测试用例
- 后续可以重复拿来回放

### 3.3 批量治理样本

- 找出可以转成用例的录制
- 批量打标签
- 批量生成用例
- 批量排除不需要的样本

### 3.4 回放验证

- 选择一个用例
- 指定目标环境
- 平台自动回放
- 看成功还是失败

### 3.5 双环境对比

- 比较两个环境的同一请求
- 看返回字段是否一致
- 检查环境配置或代码是否有差异

### 3.6 生成测试套件

- 把多个用例组合成一个套件
- 方便回归测试和发布前验证

### 3.7 定时执行

- 到指定时间自动回放
- 适合夜间回归、定时巡检

### 3.8 CI 集成

- 可以被 Jenkins、GitHub Actions、内部流水线调用
- 每次发布后自动跑回放
- 自动给 CI 回调结果

## 5. 菜单逐个讲解

### 5.1 数据总览

#### 这个菜单是干什么的

这是平台首页。

你可以先把它理解为“首页仪表盘”，主要看全局状态，不是拿来操作业务的。

#### 你在这里能看到什么

- 录制总量
- 回放总量
- 通过率趋势
- 失败任务分布
- 最近执行情况

#### 适合什么时候看

- 登录后先看一下整体状态
- 想知道最近有没有大量失败
- 想知道平台是不是已经有数据

#### 操作步骤

1. 点击左侧“数据总览”
2. 看上方统计卡片
3. 看中间的趋势图
4. 看失败分布
5. 看最近任务

#### 你可以从这里判断什么

- 平台最近是否有录制和回放
- 是否有大量失败
- 哪类任务比较活跃

### 5.2 应用管理

#### 这个菜单是干什么的

应用管理是整个系统的起点。

你先把被测系统注册进来，平台才知道要录哪个应用、回放到哪个应用、对比哪个应用。

#### 能做什么

- 新增应用
- 查看应用列表
- 编辑应用信息
- 查看应用详情
- 测试连接
- 检查目标 JVM
- 挂载 AREX Agent
- 卸载 AREX Agent
- 查看最近录制、测试用例、回放任务

#### 新手最常见的理解

这里不是在“管理业务系统本身”，而是在“管理平台认识的被测应用”。

比如：

- `pl2-bank-service`
- `vt-bank-service`

这类应用都要先在这里注册。

#### 页面怎么用

##### 1）进入应用管理列表

1. 点击左侧“应用管理”
2. 看到应用列表
3. 每一行是一套被测系统

##### 2）新增应用

1. 点击“新增应用”
2. 填应用名称
3. 填宿主机地址、用户名、端口
4. 填业务服务端口
5. 填 JVM 进程名或启动特征
6. 保存

##### 3）查看详情

1. 在列表里点“详情”
2. 进入应用详情页
3. 查看应用状态、接入信息、快捷操作、最近会话

##### 4）测试连接

1. 进入应用详情页
2. 点击“测试连接”
3. 平台会尝试连接宿主机
4. 成功说明平台能连到这台机器

##### 5）挂载 Agent

1. 先确保应用已注册
2. 确保接入信息正确
3. 点击“挂载 Agent”
4. 平台会把 AREX Agent 推到目标宿主机或 Docker 工作目录
5. 宿主机脚本模式下会修改启动参数
6. Docker Compose 模式下会生成 override 并重建容器
7. 最后让 Agent 在 JVM 里开始工作

##### 6）卸载 Agent

1. 如果你不想继续录制
2. 可以点击“卸载 Agent”
3. 平台会撤掉 Agent 挂载

##### 7）查看最近数据

应用详情里通常会看到：

- 最近会话
- 最近测试用例
- 最近回放任务

这些能帮助你快速判断这个应用是不是已经开始工作了。

##### 8）配置交易码映射

如果 SAT 和 UAT 的接口字段不一致，但交易码还是同一套，你可以在新增 / 编辑应用时直接配置交易码映射。

推荐先看这份模板文档：

- `docs/交易码映射模板.md`

填写时建议按下面的思路来：

- 一个交易码一组规则
- 请求规则和响应规则分开写
- 只做字段改名时优先用 `rename`
- 不需要的字段用 `delete`
- UAT 需要必填但 SAT 可能为空的字段用 `default`
- 需要强制改成固定值时用 `set`
- 同时保留原字段并复制新字段时用 `copy`

字段路径统一用点号写法，例如：

- `name`
- `customer.name`
- `items.0.name`
- `items.*.name`

#### 这个菜单适合做什么

- 新建被测应用
- 接入测试环境
- 确认 Agent 是否工作
- 快速跳转到录制中心、回放中心

#### 新手注意点

- 先注册应用，再录制
- 先确认宿主机连通，再挂 Agent
- 应用详情页里的信息很多，重点看“状态”和“快捷操作”

### 5.3 录制中心

#### 这个菜单是干什么的

录制中心是最核心的业务页面之一。

你可以在这里：

- 创建录制会话
- 过滤录制数据
- 查看录制结果
- 批量处理录制样本
- 从录制生成测试用例

#### 页面上通常分成几块

- 会话统计
- 查询条件
- 录制会话列表
- 样本治理视图

#### 你能做什么

- 看某次录制会话录了多少条
- 看某个会话有哪些请求
- 看请求里的交易码、场景键、URI
- 给录制样本做治理
- 批量生成测试用例

#### 操作步骤 1：新建录制会话

1. 点击“+ 新建会话”
2. 填会话名称
3. 选择所属应用
4. 填交易码过滤规则
5. 点击创建

#### 交易码过滤规则怎么填

如果你的系统是靠内部字段区分接口，比如：

- `<tx_code>A0201M14I</tx_code>`
- `<rct_code>A0201M14I</rct_code>`

你就可以在这里按规则过滤。

常见写法：

- `A0201`：前缀匹配
- `=A0201M14I`：精确匹配
- `re:^A0201.*$`：正则匹配

推荐一行一条规则。

#### 操作步骤 2：开始/停止录制

1. 先点击“开始录制”
2. 在目标系统里真实发请求
3. 录完后点击“停止录制”
4. 平台从存储服务拉录制数据
5. 数据收集完成后，录制会话下就会出现样本

#### 操作步骤 3：查看会话里的录制

1. 点击会话行
2. 打开会话详情或录制抽屉
3. 看到录制请求列表
4. 点单条录制可进入录制详情

#### 操作步骤 4：查看录制详情

录制详情里一般能看到：

- 请求信息
- 响应信息
- 交易码
- 子调用
- 录制来源

如果目标系统里发生了数据库调用，这些内容也会作为子调用的一部分展示出来。

#### 操作步骤 5：批量生成测试用例

1. 在样本治理视图里选中多个录制
2. 点击“批量生成用例”
3. 填用例前缀
4. 检查有没有重名冲突
5. 确认生成

#### 这个菜单适合做什么

- 找录制
- 过滤录制
- 给录制做治理
- 生成测试用例
- 看单条请求的详细内容

#### 新手注意点

- 录制中心不是“发起回放”的地方
- 它主要是“录完后查看和治理”
- 如果录制太多，可以先用过滤条件缩小范围

### 5.4 测试用例库

#### 这个菜单是干什么的

测试用例库是用来管理可以重复使用的测试用例。

录制只是原始数据，用例才是正式资产。

#### 能做什么

- 查看测试用例列表
- 新增测试用例
- 编辑测试用例
- 删除测试用例
- 从录制生成用例
- 加标签
- 改状态
- 导入导出
- 加入套件
- 直接发起回放

#### 操作步骤 1：查看用例列表

1. 点击“测试用例库”
2. 看用例列表
3. 可以按应用、标签、状态筛选

#### 操作步骤 2：新增测试用例

1. 点击“新增测试用例”
2. 输入名称、描述、应用
3. 粘贴或编辑请求数据
4. 保存

#### 操作步骤 3：从录制转成用例

1. 进入录制中心
2. 找到一条录制
3. 点击“生成测试用例”
4. 填用例名称
5. 保存

#### 操作步骤 4：编辑用例

1. 打开某条用例
2. 点击“编辑”
3. 修改请求参数、断言、标签、备注
4. 保存

#### 操作步骤 5：直接回放

1. 在某条用例右侧点击“回放”
2. 进入回放页面
3. 选择目标环境
4. 开始回放

#### 操作步骤 6：加入套件

1. 选中某条用例
2. 点击“加入套件”
3. 选择一个已有测试套件
4. 保存

#### 这个菜单适合做什么

- 管理回归测试资产
- 把录制变成可重复执行的用例
- 给不同业务分组
- 让测试用例沉淀下来

#### 新手注意点

- 录制和测试用例不是一回事
- 录制是“原始素材”
- 用例是“整理后的正式资产”

### 5.5 发起回放

#### 这个菜单是干什么的

回放就是把测试用例再发一遍。

这是平台最重要的验证能力之一。

#### 能做什么

- 选择一个或多个测试用例
- 选择回放目标环境
- 设置并发数
- 设置超时时间
- 设置忽略字段
- 设置断言
- 设置智能降噪
- 设置子调用 Mock
- 重复回放

#### 操作步骤

1. 点击“发起回放”
2. 选择测试用例
3. 选择目标环境
4. 展开“高级回放配置”
5. 按需要调整：
   - 并发数
   - 超时时间
   - 忽略字段
   - 断言规则
   - 智能降噪
   - 子调用 Mock
6. 点击“开始回放”

#### 回放后会发生什么

- 平台创建一个回放任务
- 后台开始执行
- 你可以在“回放历史”里看到进度
- 结束后可以看结果详情

#### 这个菜单适合做什么

- 验证某个用例在当前环境能不能跑通
- 验证改代码后有没有回归
- 验证两个环境是否一致

#### 新手注意点

- 回放不是录制
- 回放会真正访问目标服务
- 如果目标环境的数据不一致，结果也可能不一致

- 哪些任务已经完成
- 哪些任务还在运行
- 哪些任务失败了
- 哪些任务的结果要继续分析

#### 能做什么

- 查看回放任务列表
- 查看任务详情
- 看结果统计
- 导出 HTML 报告

#### 操作步骤

1. 点击“回放历史”
2. 浏览任务列表
3. 点某一条任务
4. 看任务状态、总数、成功数、失败数
5. 进入结果详情

#### 这个菜单适合做什么

- 查最近跑过什么任务
- 查失败任务
- 查某次回放的总结果

#### 新手注意点

- 这里是“回放任务”的历史
- 不要和“测试用例库”混淆

### 5.7 回放套件

#### 这个菜单是干什么的

套件就是把多个测试用例打包在一起。

它适合做批量回归。

#### 能做什么

- 新建套件
- 编辑套件
- 删除套件
- 调整用例顺序
- 选择默认目标应用
- 批量执行套件
- 查看套件运行记录

#### 操作步骤

1. 点击“回放套件”
2. 新建一个套件
3. 填写套件名称和描述
4. 把测试用例拖进去
5. 设置默认执行目标
6. 保存

#### 套件适合做什么

- 冒烟测试
- 回归测试
- 一组相关接口的统一验证

#### 新手注意点

- 套件不是用例
- 套件是“用例的集合”
- 套件通常用于批量跑

### 5.8 定时回放

#### 这个菜单是干什么的

定时回放就是按计划自动执行套件。

比如：

- 每天晚上 10 点跑一次
- 每周一早上跑一次
- 发布后自动跑一次

#### 能做什么

- 创建定时任务
- 绑定测试套件
- 设置 Cron 表达式
- 选择通知渠道
- 查看执行结果

#### 操作步骤

1. 点击“定时回放”
2. 新建任务
3. 选择一个测试套件
4. 配置 Cron 表达式
5. 设置通知方式
6. 保存

#### 这个菜单适合做什么

- 夜间巡检
- 定期回归
- 自动化健康检查

#### 新手注意点

- Cron 表达式要写对
- 套件必须先建好
- 任务只有开启后才会定时执行

### 5.9 双环境对比

#### 这个菜单是干什么的

双环境对比是拿同一条请求分别发到两个环境里比较。

典型场景：

- PL2 vs VT
- 测试环境 vs 预发环境
- 新版本 vs 旧版本

#### 能做什么

- 创建对比任务
- 设置两个目标环境
- 查看差异字段
- 看哪条请求不一致
- 导出对比报告

#### 操作步骤

1. 点击“双环境对比”
2. 新建对比任务
3. 选择左侧环境
4. 选择右侧环境
5. 选择测试用例或录制样本
6. 开始对比

#### 结果怎么看

- 如果两边一致，会显示通过
- 如果两边不一致，会显示差异字段
- 差异通常包括：
  - 返回值不同
  - 状态不同
  - 字段缺失
  - 某些数值不一致

#### 这个菜单适合做什么

- 检查两个环境是否一致
- 查版本差异
- 查参数差异

#### 新手注意点

- 对比不等于回放
- 回放是“发到一个环境”
- 对比是“同时看两个环境”

## 6. 常见操作流程示例

这里把最常用的几条流程直接写出来，照着做就行。

### 6.1 新建一个应用并挂载 Agent

1. 进入“应用管理”
2. 点击“新增应用”
3. 填写宿主机地址、用户名、端口、服务端口
4. 保存应用
5. 打开应用详情
6. 点击“测试连接”
7. 点击“挂载 Agent”
8. 等待状态变成已挂载

### 6.2 录制一次业务请求

1. 先确保应用已经挂载 Agent
2. 进入“录制中心”
3. 点击“新建会话”
4. 选择应用
5. 填交易码过滤规则
6. 保存会话
7. 点击“开始录制”
8. 在真实业务系统里操作一次请求
9. 回到录制中心点击“停止录制”
10. 等平台完成收集后进入会话详情查看录制结果

### 6.3 由录制生成测试用例

1. 在录制中心找到一条录制
2. 打开录制详情
3. 点击“生成测试用例”
4. 填写用例名称
5. 保存
6. 去测试用例库查看新生成的用例

### 6.4 发起回放

1. 进入“测试用例库”或“发起回放”
2. 选择一个用例
3. 选择目标环境
4. 设置并发和高级配置
5. 点击开始回放
6. 去“回放历史”查看结果

### 6.5 做双环境对比

1. 进入“双环境对比”
2. 创建对比任务
3. 选择左侧环境和右侧环境
4. 选择用例或录制样本
5. 开始对比
6. 查看差异报告

## 7. 常见问题

### 7.1 为什么看不到录制数据

可能原因：

- Agent 没挂上
- 会话没建
- 真实业务没跑
- 同步还没执行
- 交易码过滤太严格

### 7.2 为什么同一个接口录出来很多条

因为业务系统本身会被多次调用。

另外你们这种系统通常会用内部字段区分接口，比如：

- `tx_code`
- `rct_code`

所以同一个 URL 下面，实际可能是很多不同业务。

### 7.3 为什么回放结果和录制结果不一样

常见原因：

- 目标环境数据不一致
- 业务状态不一致
- 时间戳、流水号等动态字段没忽略
- 测试账号不同
- 数据库里已有状态变化

### 7.4 为什么双环境对比会有差异

常见原因：

- 一个环境多了一些历史数据
- 一个环境配置不同
- 两边版本不一致
- 某些字段本来就是动态字段

### 7.5 交易码过滤怎么写才稳

建议：

- 一行一条规则
- 优先用精确匹配
- 需要批量时再用前缀
- 复杂情况再写正则

示例：

```text
A0201
=A0201M14I
re:^A0201.*$
```

## 8. 最后总结

如果你只想记住一句话，可以记成：

> 这个平台的核心就是：先录真实请求，再把录制数据转成可复用的测试用例，然后用这些用例去回放、对比和回归验证。

对于新手来说，最重要的顺序是：

1. 先在应用管理把应用接进来
2. 再在录制中心建立会话，先开始录制，再停止录制
3. 然后从录制生成测试用例
4. 再去回放历史、回放套件、双环境对比里做验证
5. 最后再接入 CI 和定时任务，形成自动化流程

如果你先熟悉这五步，基本就能把这个平台用起来了。
