1. 先理解这个平台是做什么的

AREX Recorder 是一个“录制、回放、对比”平台。

它最核心的想法很简单：先把真实业务请求录下来，再把同样的请求重新发一次，最后比较两边结果是不是一致。

你可以把它理解成一个自动化回放测试平台：

  • 平时用户在系统里点按钮、提交表单、发请求。
  • 平台把这些真实请求记录下来。
  • 之后可以把这些请求拿去别的环境再跑一遍。
  • 比如拿生产录制的数据去测试环境、回归环境跑。
  • 如果结果不一样，平台会告诉你哪里不一样。
一句话理解： 它不是简单的“接口列表”，而是“把真实流量变成可重复执行的测试资产”。

2. 平台的工作原理

这个平台不是只看页面，它背后有一整条流程。

2.1 录制是怎么来的

平台支持的是 Agent 录制。

  1. 在目标 Java 应用里挂一个 AREX Agent。
  2. 用户照常访问业务系统。
  3. Agent 在 JVM 里把请求、响应、子调用记录下来。
  4. 这些记录先上传到 AREX 的存储服务。
  5. 平台再把这些数据同步回来展示。

你可以理解成：

  • 业务系统照常运行，不需要改很多代码。
  • Agent 像录音笔一样在系统内部记录发生了什么。
  • 平台负责展示、筛选、转成用例、发起回放。

2.2 录制数据里有什么

  • 请求方法，比如 GET、POST
  • 请求地址
  • 请求参数
  • 请求体
  • 响应结果
  • 耗时
  • 交易码、业务码、接口识别码
  • 子调用信息
  • 关联的录制会话

如果业务系统会访问数据库，或者调用 Redis、RPC 接口，这些也可能作为“子调用”一起被记录下来。

2.3 录制会话是什么意思

这里的录制会话不是“同步到目标系统”，而是：先点“开始录制”，让目标系统真实跑请求，再点“停止录制”，平台把存储服务里的记录拉回来。

录制会话： 先开始录制，再停止录制，平台把存储服务里的数据拉到平台数据库中，方便你在页面上看和操作。

2.4 回放是怎么来的

  1. 从录制数据里取出原始请求。
  2. 把这条请求重新发到目标环境。
  3. 取回新响应。
  4. 跟录制时的结果做比较。

2.5 对比是怎么来的

对比通常是同一批录制请求同时发到两个环境，然后比较两边响应是不是一样。比如左边发到 SIT，右边发到 UAT，平台比较两边返回。如果返回不一样，就会显示差异字段。

3. 你可以用它做哪些事

  • 回放验证
  • 双环境对比
  • 生成测试套件

4. 登录前先知道的几件事

角色能做什么
管理员能看全部菜单，能新增和修改关键配置
编辑者能创建、编辑、发起回放、管理录制
只读用户只能看数据，不能改数据
提醒： 如果是新环境，通常会有默认管理员账号。第一次登录后一定要改密码。

5. 菜单逐个讲解

5.1 数据总览

这是平台首页，先看整体状态，不是拿来做具体操作的。

  • 录制总量
  • 回放总量
  • 通过率趋势
  • 失败任务分布
  • 最近执行情况

5.2 应用管理

应用管理是整个系统的起点。你先把被测系统注册进来，平台才知道要录哪个应用、回放到哪个应用、对比哪个应用。

这里能做的事：

  • 新增应用
  • 查看应用列表
  • 编辑应用信息
  • 查看应用详情
  • 测试 SSH 连接
  • 检查目标 JVM
  • 挂载 AREX Agent
  • 卸载 AREX Agent
  • 查看最近录制、测试用例、回放任务

常见操作

  1. 点击“新增应用”，填 SSH 和服务信息。
  2. 进入详情页，先点“测试连接”。
  3. 确认没问题后点“挂载 Agent”。
  4. 在详情页查看最近会话和快捷操作。

5.3 录制中心

录制中心是最核心的业务页面之一。你可以在这里创建录制会话、过滤录制数据、查看录制结果、批量处理录制样本、从录制生成测试用例。

常见操作

  1. 点击“+ 新建会话”。
  2. 选择所属应用。
  3. 填写交易码过滤规则。
  4. 让真实业务系统跑一次请求。
  5. 回到录制中心点击“停止录制”。
  6. 在会话详情里查看录制结果。
交易码过滤建议： 你们这种按 <tx_code> 或 <rct_code> 区分接口的系统，推荐一行一条规则，比如 car001、=car001_open、re:^car001.*$。

5.4 测试用例库

测试用例库是用来管理可以重复使用的测试用例。录制只是原始数据，用例才是正式资产。

  • 查看测试用例列表
  • 新增测试用例
  • 编辑测试用例
  • 从录制生成用例
  • 加标签
  • 改状态
  • 加入套件
  • 直接发起回放

5.5 发起回放

回放就是把测试用例再发一遍。你可以选择用例、选择目标环境、设置并发、超时时间、忽略字段、断言规则、智能降噪、子调用 Mock、重复回放。

重点： 回放会真正访问目标服务，所以目标环境的数据状态会影响结果。

5.6 回放历史

这里是所有回放任务的总列表。你可以在这里看哪些任务已经完成、哪些任务还在运行、哪些任务失败了，以及结果要不要继续分析。

5.7 回放套件

套件就是把多个测试用例打包在一起，适合做批量回归、冒烟测试或者一组相关接口的统一验证。

5.8 定时回放

定时回放就是按计划自动执行套件，适合夜间巡检、定期回归、自动化健康检查。

5.9 双环境对比

双环境对比是拿同一条请求分别发到两个环境里比较。常见场景是 SIT vs UAT、测试环境 vs 预发环境、新版本 vs 旧版本。

5.10 CI 集成

CI 集成是把平台接到流水线里，代码提交后自动触发回放，回放结果自动回调给 CI。

5.11 用户管理

用户管理是给平台账号分配权限。只有管理员一般能看到。

5.12 平台指引

平台指引一般会告诉你如何配置系统、如何理解 Agent、如何设置录制和回放、常见问题怎么处理。

6. 常见操作流程示例

6.1 新建一个应用并挂载 Agent

  1. 进入“应用管理”。
  2. 点击“新增应用”。
  3. 填写 SSH 主机、用户名、端口、服务端口。
  4. 保存应用。
  5. 打开应用详情。
  6. 点击“测试连接”。
  7. 点击“挂载 Agent”。
  8. 等待状态变成已挂载。

6.2 录制一次业务请求

  1. 先确保应用已经挂载 Agent。
  2. 进入“录制中心”。
  3. 点击“新建会话”。
  4. 选择应用。
  5. 填交易码过滤规则。
  6. 保存会话。
  7. 先点击“开始录制”。
  8. 在真实业务系统里操作一次请求。
  9. 回到录制中心点击“停止录制”。
  10. 进入会话详情查看录制结果。

6.3 由录制生成测试用例

  1. 在录制中心找到一条录制。
  2. 打开录制详情。
  3. 点击“生成测试用例”。
  4. 填写用例名称。
  5. 保存。
  6. 去测试用例库查看新生成的用例。

6.4 发起回放

  1. 进入“测试用例库”或“发起回放”。
  2. 选择一个用例。
  3. 选择目标环境。
  4. 设置并发和高级配置。
  5. 点击开始回放。
  6. 去“回放历史”查看结果。

6.5 做双环境对比

  1. 进入“双环境对比”。
  2. 创建对比任务。
  3. 选择左侧环境和右侧环境。
  4. 选择用例或录制样本。
  5. 开始对比。
  6. 查看差异报告。

7. 常见问题

7.1 为什么看不到录制数据

  • Agent 没挂上
  • 会话没建
  • 真实业务没跑
  • 同步还没执行
  • 交易码过滤太严格

7.2 为什么同一个接口录出来很多条

因为业务系统本身会被多次调用。另外你们这种系统通常会用内部字段区分接口，比如 tx_code、rct_code，所以同一个 URL 下面，实际可能是很多不同业务。

7.3 为什么回放结果和录制结果不一样

  • 目标环境数据不一致
  • 业务状态不一致
  • 时间戳、流水号等动态字段没忽略
  • 测试账号不同
  • 数据库里已有状态变化

7.4 交易码过滤怎么写才稳

建议一行一条规则，优先用精确匹配，批量时用前缀，复杂情况再写正则。

car001
=car001_open
re:^car001.*$
如果你只想记住一句话，可以记成：这个平台的核心就是先录真实请求，再把录制数据转成可复用的测试资产，然后用这些资产去回放、对比和回归验证。
