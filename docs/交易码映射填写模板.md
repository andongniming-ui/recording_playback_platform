# 交易码映射填写模板

这份模板用于 SAT -> UAT 的字段适配。

适用场景：

- SAT 和 UAT 的业务逻辑一致
- 只有字段名、默认值、删除字段、枚举值这类结构差异
- 回放前需要把 SAT 请求转换成 UAT 能接受的格式
- 回放后需要把 UAT 响应归一化后再做比对

> 建议一条交易码一组配置。  
> 如果一个交易码没有变化，就不用填。  
> 如果一个交易码有多个字段变化，就在同一组配置里把规则一次写全。

---

## 1. 你们需要提供什么

每个有变化的交易码，至少提供下面几类信息：

1. 交易码编号
2. SAT 原始请求样例
3. UAT 目标请求样例
4. 请求字段映射
5. 响应字段映射
6. 默认值补齐规则
7. 需要删除的字段
8. 特殊说明

---

## 3. 平台支持的规则类型

当前平台支持下面几类规则：

| 规则类型 | 含义 | 典型用途 |
|---|---|---|
| `rename` | 改字段名 | `name -> cst_name` |
| `delete` | 删除字段 | 删除 `debug_flag` |
| `default` | 字段为空时补默认值 | `branch_code = 0101` |
| `set` | 强制赋值 | 无论原值是什么都改成固定值 |
| `copy` | 复制字段 | 保留原字段，再生成一份新字段 |

### 规则填写说明

- `rename`
  - 必填：`source`、`target`
  - 作用：把源字段改成目标字段

- `delete`
  - 必填：`source`
  - 作用：删除该字段

- `default`
  - 必填：`source`、`value`
  - 作用：字段为空时才补默认值

- `set`
  - 必填：`source`、`value`
  - 作用：不管原值是什么，都改成指定值

- `copy`
  - 必填：`source`、`target`
  - 建议：同时写 `value` 作为兜底
  - 作用：复制一份字段到新字段，原字段保留

## 4. 字段路径怎么写

字段路径建议使用点号写法。

### 常见写法

- `name`
- `customer.name`
- `items.0.name`
- `items.*.name`
- `*.name`

### 说明

- `name`：直接访问根字段
- `customer.name`：访问嵌套对象
- `items.0.name`：访问列表中第 0 个元素
- `items.*.name`：对列表中每一项生效
- `*.name`：对根对象下每个分支尝试匹配

> XML 报文如果只有一个根节点，例如 `<request>...</request>`，  
> 平台会优先识别根节点内的实际业务字段，所以一般直接写 `name`、`branch_code` 这类路径即可。

## 5. 单个交易码的完整示例

### 5.1 交易码

```text
car001_open
```

### 5.2 SAT 原始请求

```xml
<request>
  <transaction_id>car001_open</transaction_id>
  <SYS_EVT_TRACE_ID>1234567890123456789012345</SYS_EVT_TRACE_ID>
  <name>张三</name>
  <idNo>110101199001011234</idNo>
  <branch_code></branch_code>
  <debug_flag>1</debug_flag>
</request>
```

### 5.3 UAT 目标请求

```xml
<request>
  <transaction_id>car001_open</transaction_id>
  <SYS_EVT_TRACE_ID>1234567890123456789012345</SYS_EVT_TRACE_ID>
  <cst_name>张三</cst_name>
  <cert_no>110101199001011234</cert_no>
  <branch_code>0101</branch_code>
</request>
```

### 5.4 请求映射

| source_field | target_field | rule_type | value | remark |
|---|---|---|---|---|
| name | cst_name | rename |  | 姓名改名 |
| idNo | cert_no | rename |  | 证件号改名 |
| branch_code | branch_code | default | 0101 | 默认分行号 |
| debug_flag |  | delete |  | 删除调试字段 |

### 5.5 响应映射

| source_field | target_field | rule_type | value | remark |
|---|---|---|---|---|
| cst_name | name | rename |  | 回放后统一字段名 |
| debug_flag |  | delete |  | 删除调试字段 |

## 6. 推荐提交格式

如果你们直接要交给平台导入，建议提交成 JSON 数组。

### JSON 示例

```json
[
  {
    "transaction_code": "car001_open",
    "enabled": true,
    "description": "基础资料字段映射",
    "request_rules": [
      { "type": "rename", "source": "name", "target": "cst_name" },
      { "type": "rename", "source": "idNo", "target": "cert_no" },
      { "type": "default", "source": "branch_code", "value": "0101" },
      { "type": "delete", "source": "debug_flag" }
    ],
    "response_rules": [
      { "type": "rename", "source": "cst_name", "target": "name" },
      { "type": "delete", "source": "debug_flag" }
    ]
  }
]
```

## 7. 填写检查清单

提交前建议检查：

- 交易码是否写对
- 请求规则和响应规则是否分开
- `rename` 是否同时填写了 `source` 和 `target`
- `delete` 是否只保留 `source`
- `default` / `set` 是否填写了 `value`
- 字段路径是否用了点号写法
- 是否把调试字段删掉了
- 是否把需要固定值的字段补齐了

---

## 8. 什么情况要单独提出来

如果出现下面这些情况，说明不只是简单字段改名：

- 一个字段拆成多个字段
- 多个字段合成一个字段
- 字段层级变化
- 日期格式变化
- 金额单位变化
- 枚举值变化
- 需要复杂计算

这种交易码建议单独备注，不要直接塞进简单的 rename 表里。
