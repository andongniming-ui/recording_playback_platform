# 交易码映射模板

这份模板用于 SAT -> UAT 的字段适配。  
适用场景：

- SAT 和 UAT 的接口交易码一致，但字段名不一致
- SAT 录制后，UAT 回放前需要按交易码把请求改成 UAT 能识别的格式
- UAT 回放响应回来后，需要先归一化，再和 SAT 结果比对

> 当前平台的映射规则是“交易码级”的。也就是说，一个交易码配一组请求规则 + 响应规则。  
> 规则存放在应用配置里，平台回放时会自动按交易码加载。

---

## 1. 你们要提供什么

建议你们按“表格”先填，再由平台转换成 JSON 配置。  
每条记录最少要说明：

- 交易码
- 规则方向：请求 / 响应
- 源字段路径
- 目标字段路径
- 规则类型
- 默认值或固定值
- 是否启用
- 备注

### 推荐的表格模板

| transaction_code | direction | source_field | target_field | rule_type | value | enabled | remark |
|---|---|---|---|---|---|---|---|
| A0201M14I | request | name | cst_name | rename |  | true | 开户请求字段改名 |
| A0201M14I | request | idNo | cert_no | rename |  | true | 证件号改名 |
| A0201M14I | request | branch_code | branch_code | default | 0101 | true | 分行号补默认值 |
| A0201M14I | request | debug_flag |  | delete |  | true | 调试字段删除 |

## 2. 平台最终吃的配置长什么样

平台应用配置里的 `transaction_mappings` 最终是一个 JSON 数组。  
每个交易码一组配置，结构如下：

```json
[
  {
    "transaction_code": "A0201M14I",
    "enabled": true,
    "description": "开户字段映射",
    "request_rules": [
      { "type": "rename", "source": "name", "target": "cst_name" },
      { "type": "rename", "source": "idNo", "target": "cert_no" },
      { "type": "default", "source": "branch_code", "value": "0101" },
      { "type": "delete", "source": "debug_flag" }
    ],
    "response_rules": [
      { "type": "rename", "source": "cst_name", "target": "name" },
      { "type": "delete", "source": "debug_flag" }
    ]
  }
]
```

### 顶层字段说明

| 字段 | 说明 |
|---|---|
| `transaction_code` | 交易码，必须唯一，例如 `A0201M14I` |
| `enabled` | 是否启用，默认 `true` |
| `description` | 说明文字，方便后续维护 |
| `request_rules` | 回放到 UAT 之前，对请求做的转换规则 |
| `response_rules` | 回放后，对 UAT 响应做的归一化规则 |

---

## 3. 规则类型说明

当前平台支持 5 种规则类型。

| 规则类型 | 含义 | 典型用途 |
|---|---|---|
| `rename` | 改字段名 | `name -> cst_name` |
| `delete` | 删除字段 | 删除 `debug_flag`、`trace_flag` |
| `default` | 字段为空时补默认值 | `branch_code = 0101` |
| `set` | 强制赋值 | 无论原值是什么，都改成指定值 |
| `copy` | 复制到新字段 | 原字段保留，同时生成一份新字段 |

### 规则字段说明

| 字段 | 说明 |
|---|---|
| `type` | 规则类型，必填 |
| `source` | 源字段路径，必填 |
| `target` | 目标字段路径，`rename` / `copy` 时必填 |
| `value` | 默认值或固定值，`default` / `set` / `copy` 场景使用 |
| `enabled` | 是否启用，默认 `true` |
| `description` | 备注说明 |

---

## 4. 字段路径怎么写

平台的字段路径用点号写法。

### 常见写法

- `name`
- `customer.name`
- `items.0.name`
- `items.*.name`
- `*.name`

### 含义说明

- `name`：直接访问根字段
- `customer.name`：访问嵌套对象
- `items.0.name`：访问列表第 0 个元素
- `items.*.name`：对列表中每一项都生效
- `*.name`：对根对象下的每个分支都尝试匹配

> 如果你的 XML 请求外层只有一个根节点，比如 `<request>...</request>`，  
> 平台会优先识别里面的实际业务字段，所以像 `name`、`branch_code` 这种路径一般可以直接写，不需要手工加根节点。

## 5. 推荐填写方式

### 方式一：简单改名

适合绝大多数字段改名。

```json
{
  "type": "rename",
  "source": "name",
  "target": "cst_name"
}
```

### 方式二：删除无关字段

适合调试字段、流水字段、临时字段。

```json
{
  "type": "delete",
  "source": "debug_flag"
}
```

### 方式三：补默认值

适合 UAT 需要必填，但 SAT 录制时可能为空的字段。

```json
{
  "type": "default",
  "source": "branch_code",
  "value": "0101"
}
```

### 方式四：强制赋值

适合一些 UAT 环境固定要求的字段。

```json
{
  "type": "set",
  "source": "channel_code",
  "value": "UAT"
}
```

### 方式五：复制字段

适合 UAT 既要保留原字段，又要多一份新字段。

```json
{
  "type": "copy",
  "source": "name",
  "target": "cst_name"
}
```

## 6. 示例：A0201M14I 开户交易

### SAT 原始请求

```xml
<request>
  <transaction_id>A0201M14I</transaction_id>
  <SYS_EVT_TRACE_ID>1234567890123456789012345</SYS_EVT_TRACE_ID>
  <name>张三</name>
  <idNo>110101199001011234</idNo>
  <branch_code></branch_code>
  <debug_flag>1</debug_flag>
</request>
```

### UAT 需要的字段

```xml
<request>
  <transaction_id>A0201M14I</transaction_id>
  <SYS_EVT_TRACE_ID>1234567890123456789012345</SYS_EVT_TRACE_ID>
  <cst_name>张三</cst_name>
  <cert_no>110101199001011234</cert_no>
  <branch_code>0101</branch_code>
</request>
```

### 配置示例

```json
[
  {
    "transaction_code": "A0201M14I",
    "enabled": true,
    "description": "开户交易字段映射",
    "request_rules": [
      { "type": "rename", "source": "name", "target": "cst_name" },
      { "type": "rename", "source": "idNo", "target": "cert_no" },
      { "type": "default", "source": "branch_code", "value": "0101" },
      { "type": "delete", "source": "debug_flag" }
    ],
    "response_rules": [
      { "type": "rename", "source": "cst_name", "target": "name" },
      { "type": "delete", "source": "debug_flag" }
    ]
  }
]
```

## 7. 建议的填写顺序

1. 先确认交易码是否需要映射
2. 再确认请求字段改了什么
3. 再确认响应字段改了什么
4. 最后补默认值、删除字段、固定值
5. 一个交易码先配一版，回放后再补复杂规则

---

## 8. 填写检查清单

提交映射表前，先检查这几项：

- 交易码是否写对
- 请求规则和响应规则有没有分开
- `rename` 是否同时配了 `source` 和 `target`
- `delete` 是否只填了 `source`
- `default` / `set` 是否填了 `value`
- 路径是否用了平台支持的点号写法
- 是否把调试字段、流水字段这类噪音字段删掉
- 是否把需要固定值的字段补齐了

---

## 9. 什么时候该考虑代码扩展

如果出现下面这些情况，说明单纯的配置不够了：

- 一个字段拆成多个字段
- 多个字段合成一个字段
- 需要复杂计算
- 需要按条件分支
- 需要根据数据库返回值动态补字段

这种交易码先不要强行塞进配置，建议单独做扩展实现。
