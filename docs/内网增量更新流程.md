# 内网增量更新流程

这份文档是日常外网开发、内网离线部署时的固定操作流程。目标是每次只传增量包，不再全量传整个项目。

## 核心原则

增量包是按 Git 差异生成的：

```bash
当前外网代码 - 云内已部署基线 = 本次增量包
```

所以最重要的是记住“云内当前部署版本对应的外网 commit”。每次云内更新成功后，以增量包里的 `manifest.env` 中的 `HEAD_COMMIT` 作为下一次的 `--base`。

不要直接用云内机器上的 `git rev-parse HEAD` 当新基线，因为增量包只是覆盖文件，不一定会更新云内 `.git` 的 HEAD。

## 外网开发机

### 1. 修改代码并完成自测

按正常方式修改代码。上传前建议至少跑一遍相关验证，例如：

```bash
cd /home/recording_playback_platform

cd platform/frontend
npm run build

cd /home/recording_playback_platform
python3 -m py_compile platform/backend/api/v1/applications.py platform/backend/schemas/application.py
pytest platform/backend/tests/test_applications.py -q
```

具体跑哪些测试可以按本次改动范围调整。前端改动至少跑 `npm run build`。

### 2. 确认本次基线

基线就是“云内当前正在运行的版本”对应的外网 commit。

如果上次更新后已经记录过 `HEAD_COMMIT`，直接用那个值。假设上次记录的是：

```text
64cc16b5
```

那么这次生成增量包时就用它作为 `--base`。

如果忘了上次基线，可以先看最近提交辅助判断：

```bash
git log --oneline -10
```

### 3. 生成增量包

在外网项目根目录执行：

```bash
cd /home/recording_playback_platform
scripts/create-intranet-incremental.sh --base 64cc16b5 --out-dir /mnt/e/test/ceshi
```

输出类似：

```text
[OK] Created /mnt/e/test/ceshi/intranet-incremental-20260509-153000.tar.gz
[OK] Size: 36K
[OK] Changed files: 5
[OK] Deleted files: 1
```

Windows 上对应路径一般是：

```text
E:\test\ceshi\intranet-incremental-20260509-153000.tar.gz
```

### 4. 传输到云内

只需要把生成的 `.tar.gz` 文件传到云内服务器。

不要传 `node_modules`，也不要手动复制整个项目目录。

## 云内部署机

### 5. 解压增量包

进入你上传增量包的目录，执行：

```bash
tar -xzf intranet-incremental-20260509-153000.tar.gz
```

解压后会出现同名目录：

```text
intranet-incremental-20260509-153000/
```

### 6. 应用增量包

假设云内项目部署目录是：

```text
/home/ap/recording_playback_platform
```

执行：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /home/ap/recording_playback_platform
```

脚本会自动完成：

- 备份即将被覆盖或删除的旧文件
- 覆盖本次变更文件
- 删除外网已经删除的文件
- 自动判断是否需要重建 `backend` 或 `frontend` 容器
- 执行 `docker compose up -d --build` 重建对应服务

备份文件会放在：

```text
/home/ap/recording_playback_platform/runtime/intranet-update-backups/
```

### 7. 检查服务状态

应用脚本结束后会输出 `docker compose ps`。如果想手动再看一遍：

```bash
cd /home/ap/recording_playback_platform/deploy/intranet
docker compose --env-file .env.production -f docker-compose.intranet.yml ps
```

如果页面或接口异常，再看日志：

```bash
docker compose --env-file .env.production -f docker-compose.intranet.yml logs -f backend
docker compose --env-file .env.production -f docker-compose.intranet.yml logs -f frontend
```

### 8. 如需回滚

应用增量包时，脚本会先备份即将被覆盖或删除的旧文件。日志里会出现类似输出：

```text
[INFO] Backup written: /home/ap/recording_playback_platform/runtime/intranet-update-backups/20260509-153000.tar.gz
```

如果更新后发现有问题，先找到这两个东西：

```bash
TARGET=/home/ap/recording_playback_platform
PKG=/path/to/intranet-incremental-20260509-153000
BACKUP=/home/ap/recording_playback_platform/runtime/intranet-update-backups/20260509-153000.tar.gz
```

完整回滚执行：

```bash
tmpdir="$(mktemp -d)"

# 1. 找出本次新增的文件。备份包里没有的 changed-files，就是本次新增文件。
tar -tzf "$BACKUP" | sed 's#^\./##' | sort -u > "$tmpdir/backup-files.txt"
sort -u "$PKG/changed-files.txt" > "$tmpdir/changed-files.txt"

# 2. 删除本次新增文件。
comm -23 "$tmpdir/changed-files.txt" "$tmpdir/backup-files.txt" | while IFS= read -r rel; do
  [ -n "$rel" ] && rm -rf "$TARGET/$rel"
done

# 3. 恢复被覆盖或被删除的旧文件。
tar -xzf "$BACKUP" -C "$TARGET"

rm -rf "$tmpdir"
```

回滚完代码后，重建服务：

```bash
cd /home/ap/recording_playback_platform/deploy/intranet
docker compose --env-file .env.production -f docker-compose.intranet.yml up -d --build backend frontend
docker compose --env-file .env.production -f docker-compose.intranet.yml ps
```

如果应用增量包时日志显示：

```text
[INFO] No existing target files needed backup.
```

说明这次只新增了文件，没有覆盖旧文件。此时回滚只需要删除本次增量包里的 `changed-files.txt` 文件清单：

```bash
TARGET=/home/ap/recording_playback_platform
PKG=/path/to/intranet-incremental-20260509-153000

while IFS= read -r rel; do
  [ -n "$rel" ] && rm -rf "$TARGET/$rel"
done < "$PKG/changed-files.txt"
```

回滚成功后，不要记录本次增量包的 `HEAD_COMMIT` 作为新基线。继续沿用回滚前的旧基线。

### 9. 记录下一次基线

更新成功后，查看本次增量包里的 `manifest.env`：

```bash
cat intranet-incremental-20260509-153000/manifest.env
```

重点记录这一行：

```text
HEAD_COMMIT=xxxxxxx
```

把这个 `HEAD_COMMIT` 记下来。下一次外网生成增量包时，把它作为 `--base`。

例如本次 `manifest.env` 里是：

```text
HEAD_COMMIT=abc1234
```

那么下次外网生成增量包就用：

```bash
scripts/create-intranet-incremental.sh --base abc1234 --out-dir /mnt/e/test/ceshi
```

## 常用可选参数

只复制文件，不重建容器：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /home/ap/recording_playback_platform --rebuild none
```

强制只重建后端：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /home/ap/recording_playback_platform --rebuild backend
```

强制只重建前端：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /home/ap/recording_playback_platform --rebuild frontend
```

强制重建前后端：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /home/ap/recording_playback_platform --rebuild all
```

## 打包规则说明

默认会打进增量包：

- 从 `--base` 到当前工作区的已修改文件
- 新增文件
- 删除文件清单
- 未被 `.gitignore` 忽略的 untracked 文件
- 云内应用脚本 `apply-incremental-update.sh`
- 包信息 `manifest.env`

默认不会打进增量包：

- `.git`
- `node_modules`
- Python 虚拟环境
- 日志、缓存
- 数据库数据
- SSH key
- `.env`
- `deploy/intranet/.env.production`
- `runtime/offline-bundle`

如果确实要同步真实环境配置，可以生成包时加：

```bash
scripts/create-intranet-incremental.sh --base 64cc16b5 --out-dir /mnt/e/test/ceshi --include-env
```

一般不建议这么做，除非明确知道这次要覆盖云内环境配置。

## 依赖变更注意事项

如果本次改了这些文件：

```text
platform/requirements.txt
platform/frontend/package.json
platform/frontend/package-lock.json
```

云内离线构建可能还需要同步新的离线依赖包或镜像缓存。普通 Python 代码、Vue 页面、部署脚本、配置模板修改，一般只需要增量包即可。
