# 内网增量更新流程

这个流程用于外网开发、内网离线部署的日常小改动。目标是只传变更文件，而不是每次传全量部署包。

## 外网生成增量包

先确认内网上一次部署对应的 Git 提交号或标签，例如 `2738f312`。然后在外网项目根目录执行：

```bash
scripts/create-intranet-incremental.sh --base 2738f312 --out-dir /mnt/e/test/ceshi
```

生成结果类似：

```text
E:\test\ceshi\intranet-incremental-20260509-153000.tar.gz
```

默认会包含：

- 从 `--base` 到当前工作区的已修改、已新增文件
- 已删除文件清单
- 内网应用脚本 `apply-incremental-update.sh`
- untracked 但未被 `.gitignore` 忽略的新文件

默认不会包含：

- `.env`、`deploy/intranet/.env.production` 等真实环境配置
- 数据库数据、上传数据、ssh key
- 日志、缓存、`.git`
- `runtime/offline-bundle` 这类离线依赖大包

如果确实要同步真实环境配置，显式加 `--include-env`，但一般不建议。

## 内网应用增量包

把 `.tar.gz` 传到内网服务器后，在内网服务器执行：

```bash
tar -xzf intranet-incremental-20260509-153000.tar.gz
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /path/to/recording_playback_platform
```

脚本会：

- 备份即将被覆盖或删除的旧文件到 `runtime/intranet-update-backups`
- 覆盖变更文件
- 删除外网已经删除的文件
- 自动判断是否需要重建 `backend` 或 `frontend`

只复制文件、不重建容器：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /path/to/recording_playback_platform --rebuild none
```

强制重建指定服务：

```bash
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /path/to/recording_playback_platform --rebuild backend
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /path/to/recording_playback_platform --rebuild frontend
bash intranet-incremental-20260509-153000/apply-incremental-update.sh /path/to/recording_playback_platform --rebuild all
```

## 依赖变更注意事项

如果改了这些文件，内网离线构建可能还需要刷新离线依赖包：

- `platform/requirements.txt`
- `platform/frontend/package.json`
- `platform/frontend/package-lock.json`

普通 Python 业务代码、Vue 页面、部署脚本修改，一般只需要增量包即可。

## 建议的基线管理

每次内网更新成功后，记录本次外网 Git 提交号：

```bash
git rev-parse --short HEAD
```

下一次生成增量包时，把这个提交号作为 `--base`。

如果当次修改没有提交，也可以先用当前基线生成包；脚本会把工作区里的未提交改动一起打进去。
