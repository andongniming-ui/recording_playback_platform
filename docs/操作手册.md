## 1. 系统概述

AREX（Automated Record and rEplay eXecutor）是一个基于 Java Agent 的流量录制回放框架，可以无侵入地捕获 Java/Spring Boot 服务的 HTTP 请求及其下游调用（MySQL、Redis 等），并在回放时自动 Mock 这些子调用，从而实现真实流量驱动的接口测试。

**AREX Recorder** 是在 AREX 官方组件之上构建的完整管理平台。相比直接使用 AREX 官方 UI，本平台的核心价值在于：一是提供了完整的 Agent 生命周期管理（SSH 远程挂载/卸载，无需登录目标服务器）；二是内置了基于 deepdiff 的智能差异对比引擎，支持 JSONPath 忽略规则和 30+ 动态字段自动降噪；三是提供了团队协作所需的用户权限体系（RBAC）、测试套件、定时任务和 CI/CD 集成能力，可以直接对接企业的 Jenkins 或 GitHub Actions 流水线。

**适用场景：**
- 对 Java/Spring Boot + MySQL 技术栈的服务进行回归测试
- 通过录制生产流量构建高覆盖度的接口测试用例库
- 在 CI/CD 流水线中自动化执行回放验证，确保每次发布不引入回归缺陷
- 团队内多人协作管理测试用例，按应用和标签分类维护

## 2. 系统架构

### 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                  目标 Spring Boot 服务 (JDK 8)               │
│                  localhost:8080 / 生产服务器                  │
└────────────────────────┬────────────────────────────────────┘
                         │ JVM Agent 注入（-javaagent）
                         ▼
              ┌──────────────────────┐
              │   arex-agent.jar     │  ← SSH/SFTP 上传
              │  （流量录制 Hook）    │     由 arex-recorder 完成
              └──────────┬───────────┘
                         │ HTTP 上报录制数据（batchSaveMockers）
                         ▼
              ┌──────────────────────┐
              │  arex-storage 服务   │  官方存储组件
              │  默认 :8093          │  MongoDB + 官方协议
              └──────────┬───────────┘
                         │ REST API 查询录制数据
                         ▼
┌────────────────────────────────────────────────────────────┐
│            arex-recorder backend (FastAPI / Python 3.11)   │
│                                                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 应用管理  │  │ 录制同步  │  │ 回放引擎  │  │ 定时任务  │  │
│  │ SSH执行器 │  │ AREX代理  │  │ 差异对比  │  │APScheduler│ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
│                    SQLite / MySQL 8.0                       │
└───────────────────────┬────────────────────────────────────┘
                        │ REST API / WebSocket
                        ▼
┌────────────────────────────────────────────────────────────┐
│         arex-recorder frontend (Vue3 + Naive UI)           │
│              http://localhost:5173                          │
└────────────────────────────────────────────────────────────┘
```

### 各组件职责

| 组件 | 职责 |
|------|------|
| **arex-agent.jar** | 注入目标 JVM，在运行时 Hook HTTP 入口和 MySQL/Redis 等下游调用，将流量录制到 arex-storage |
| **arex-storage** | AREX 官方存储服务，负责持久化录制数据，提供查询接口 |
| **backend/integration/arex_proxy** | 兼容 AREX agent 上报协议（接收 zstd 压缩的 `batchSaveMockers` 请求并转发） |
| **backend/integration/ssh_executor** | 基于 Paramiko 实现 SSH 连接、SFTP 上传 Agent JAR、修改启动脚本注入 `-javaagent` 参数 |
| **backend/core/replay_executor** | 异步并发回放引擎，从数据库读取测试用例，向目标服务发送请求，收集响应并调用 diff/assertion 引擎 |
| **backend/core/scheduler** | 基于 APScheduler 的 AsyncIOScheduler，管理 Cron 定时任务，到期后自动触发套件回放 |
| **backend/utils/diff** | 基于 deepdiff 的差异对比引擎，支持 ignore/numeric_tolerance/regex_match/type_only 四类规则 |
| **backend/utils/assertions** | 自定义断言评估引擎，支持 7 种断言类型 |
| **backend/utils/notify** | 通知推送，支持钉钉（DingTalk）和企业微信（WeCom）Webhook |

### 数据流向

**录制流：**
目标服务接收真实请求 → arex-agent 拦截并序列化请求/响应/子调用 → HTTP 上报到 arex-storage → 用户在平台点击"Sync"按钮 → backend 调用 arex-storage REST API 拉取数据 → 存入本地 `recording` 表

**回放流：**
用户选择测试用例（或整个套件）→ 创建 `ReplayJob` → replay_executor 按并发度异步发送 HTTP 请求到目标服务 → 收到响应后调用 diff 引擎对比 expected_response → 评估 assertion 规则 → 写入 `ReplayResult` → 通过 WebSocket 实时推送进度到前端

### 技术选型说明

- **FastAPI + SQLAlchemy async**：FastAPI 原生支持 async/await，配合 async SQLAlchemy 可以在单进程内高效处理并发回放请求和 WebSocket 连接，无需引入消息队列。
- **Vue3 + Naive UI**：Vue3 Composition API 使复杂的状态逻辑（如 WebSocket 进度订阅）更易维护；Naive UI 提供了开箱即用的 JSON 编辑器集成接口，适合用 Monaco Editor 展示和编辑用例数据。
- **SQLite / MySQL 双模**：开发环境使用 SQLite 零配置启动，生产环境切换 MySQL 8.0 只需修改一个环境变量，无需改代码。

## 3. 安装与部署

### 3.1 环境要求

| 依赖 | 最低版本 | 说明 |
|------|----------|------|
| Python | 3.11+ | 后端运行环境 |
| Node.js | 18+ | 前端构建环境 |
| Docker | 20+ | 用于运行 MySQL（可选） |
| Docker Compose | 2.0+ | 一键部署 |
| arex-storage | 官方最新版 | 需单独部署，提供录制数据存储 |

目标服务需满足：JDK 8、Spring Boot 应用、SSH 可达（用于 Agent 挂载）。

### 3.2 开发环境搭建

```bash
# 第一步：克隆仓库
git clone <repo-url> arex-recorder
cd arex-recorder

# 第二步：复制并编辑配置文件
cp .env.example .env
# 至少修改 AR_SECRET_KEY，开发环境可保持 AR_DB_TYPE=sqlite

# 第三步：启动 MySQL（如需使用 MySQL 而非 SQLite）
docker-compose up -d db

# 第四步：安装后端 Python 依赖
pip install -r requirements.txt

# 第五步：启动后端（自动建表、自动创建默认管理员）
uvicorn backend.main:app --reload --host 0.0.0.0 --port 8000

# 第六步：安装前端依赖并启动开发服务器
cd frontend
npm install
npm run dev
```

启动完成后访问 http://localhost:5173，使用默认账号 `admin` / `admin123` 登录。

> **重要：首次登录后请立即修改默认密码。**

### 3.3 生产环境 Docker Compose 部署

```bash
# 第一步：修改 .env 为生产配置
cat >> .env << 'EOF'
AR_DB_TYPE=mysql
AR_DB_URL=mysql+aiomysql://arex:yourpassword@db:3306/arex_recorder
AR_SECRET_KEY=$(openssl rand -hex 32)
AR_DEBUG=false
AR_CORS_ORIGINS=["https://your-domain.com"]
EOF

# 第二步：构建镜像并启动所有服务
docker-compose up -d --build

# 第三步：查看后端日志确认启动成功
docker-compose logs -f backend

# 第四步：验证服务健康状态
curl http://localhost:8000/api/health
# 返回 {"status": "ok", "version": "1.0.0"} 表示正常
```

**服务端口汇总：**

| 服务 | 地址 |
|------|------|
| 前端 | http://localhost:5173 |
| 后端 API | http://localhost:8000 |
| API 文档（Swagger） | http://localhost:8000/docs |
| API 文档（ReDoc） | http://localhost:8000/redoc |
| MySQL | localhost:3307 |

**生产安全注意事项：**
- 必须将 `AR_SECRET_KEY` 替换为随机字符串（使用 `openssl rand -hex 32` 生成）
- 建议在前端配置 Nginx 反向代理并启用 HTTPS
- SSH 私钥文件权限必须设为 `600`：`chmod 600 ssh_keys/*`
- `AR_CORS_ORIGINS` 应配置为实际域名，不要在生产中使用 `["*"]`

### 3.4 配置项详解

所有配置通过 `.env` 文件或环境变量注入，变量名均以 `AR_` 为前缀：

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `AR_DB_TYPE` | `sqlite` | 数据库类型，可选 `sqlite` 或 `mysql` |
| `AR_DB_URL` | SQLite 文件路径 | 完整数据库连接字符串；MySQL 格式：`mysql+aiomysql://user:pass@host:3306/dbname` |
| `AR_SECRET_KEY` | `changeme-in-production` | JWT 签名密钥，**生产环境必须修改** |
| `AR_AREX_STORAGE_URL` | `http://localhost:8093` | arex-storage 服务地址，可被应用级配置覆盖 |
| `AR_AREX_AGENT_JAR_PATH` | `/home/test/arex-agent/arex-agent.jar` | 本地 AREX Agent JAR 文件路径，用于 SSH 上传 |
| `AR_SSH_KEYS_DIR` | `./ssh_keys` | SSH 私钥文件存储目录 |
| `AR_ACCESS_TOKEN_EXPIRE_MINUTES` | `60` | JWT Access Token 有效期（分钟） |
| `AR_REFRESH_TOKEN_EXPIRE_DAYS` | `7` | JWT Refresh Token 有效期（天） |
| `AR_DEFAULT_REPLAY_CONCURRENCY` | `5` | 新建回放任务的默认并发数 |
| `AR_DEFAULT_REPLAY_TIMEOUT_MS` | `5000` | 新建回放任务的默认超时时间（毫秒） |
| `AR_CORS_ORIGINS` | `["*"]` | 允许的跨域来源，JSON 数组格式 |
| `AR_DEBUG` | `false` | 是否开启调试模式（开启后 FastAPI 返回详细错误堆栈） |

### 3.5 首次启动初始化

系统在启动时会自动执行以下初始化操作（`main.py` lifespan 事件）：

1. 调用 `init_db()` 创建所有数据库表（如表已存在则跳过）
2. 检测 `user` 表是否为空，若为空则自动创建默认管理员账号：
   - 用户名：`admin`
   - 密码：`admin123`
   - 角色：`admin`
3. 从数据库加载所有已激活的定时任务并注册到 APScheduler

**首次登录后的必要操作：**
1. 登录后进入"系统设置"，确认 `AR_AREX_STORAGE_URL` 指向正确的 arex-storage 地址
2. 在"用户管理"中修改管理员密码
3. 在"应用管理"中创建第一个应用

## 4. 功能模块详解

### 4.1 用户与权限管理

#### 功能简介

系统采用基于角色的访问控制（RBAC），所有 API 接口均通过 JWT Bearer Token 鉴权。用户角色分为三个等级：`admin`（管理员）、`editor`（编辑者）、`viewer`（只读用户），角色权限从高到低递减。

#### 使用场景

- 创建团队成员账号并分配合适的操作权限
- 给外部监控系统或只读展示大屏分配 `viewer` 账号
- 管理员进行用户停用和密码重置操作

#### 三种角色权限对比

| 操作 | Admin | Editor | Viewer |
|------|-------|--------|--------|
| 查看所有数据（应用、用例、报告） | ✓ | ✓ | ✓ |
| 创建/编辑应用 | ✓ | ✓ | ✗ |
| 挂载/卸载 Agent | ✓ | ✓ | ✗ |
| 创建/编辑测试用例 | ✓ | ✓ | ✗ |
| 触发回放、套件执行 | ✓ | ✓ | ✗ |
| 创建/编辑定时任务 | ✓ | ✓ | ✗ |
| 删除应用（硬删除） | ✓ | ✗ | ✗ |
| 用户管理（创建/删除用户） | ✓ | ✗ | ✗ |
| CI Token 管理 | ✓ | ✗ | ✗ |

#### 操作步骤

**创建用户（Admin 操作）：**

1. 登录后点击左侧导航"用户管理"
2. 点击"新建用户"按钮
3. 填写用户名（唯一，不可重复）、初始密码
4. 选择角色：`admin` / `editor` / `viewer`
5. 点击"确认"，系统创建账号并激活

**修改密码（用户自行操作）：**

1. 点击右上角头像，选择"修改密码"
2. 输入当前密码和新密码（两次确认）
3. 提交后当前 Token 立即失效，需重新登录

**停用用户（Admin 操作）：**

1. 在用户列表找到目标用户
2. 点击"停用"按钮，`is_active` 字段置为 `false`
3. 该用户下次请求时将返回 401，已颁发的 Token 仍有效直到过期

#### JWT Token 使用方式

登录成功后 API 返回：
```json
{
  "access_token": "eyJhbGci...",
  "refresh_token": "eyJhbGci...",
  "token_type": "bearer"
}
```

后续所有 API 请求需在 Header 中携带：
```
Authorization: Bearer eyJhbGci...
```

Access Token 默认 60 分钟过期，过期后使用 Refresh Token 换取新的 Access Token：
```bash
curl -X POST http://localhost:8000/api/v1/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{"refresh_token": "eyJhbGci..."}'
```

#### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `username` | string | 用户名，全局唯一，长度上限 128 字符 |
| `role` | string | 角色：`admin` / `editor` / `viewer` |
| `is_active` | bool | 账号是否激活，`false` 时禁止登录 |
| `hashed_password` | string | bcrypt 哈希存储，不可逆 |

#### 注意事项

- 系统至少需要保留一个 `admin` 账号，不要将所有管理员同时停用
- 删除用户为物理删除，其创建的用例、任务等数据不会级联删除
- JWT 密钥（`AR_SECRET_KEY`）一旦修改，所有在线用户的 Token 立即失效

### 4.2 应用管理与 Agent 挂载

#### 功能简介

应用是平台的核心管理单元，对应一个目标 Java/Spring Boot 服务实例。在应用中配置 SSH 连接信息后，平台可以自动完成 AREX Agent JAR 上传、`-javaagent` 参数注入和进程重启的全套流程，无需手动登录目标服务器操作。

#### 使用场景

- 将一个新的 Java 微服务接入录制回放体系
- 在不同环境（测试环境、预发布环境）分别注册同一个服务的不同实例
- 调整某个服务的录制采样率，控制录制数据量

#### 操作步骤

**注册新应用：**

1. 点击左侧"应用管理" → "新建应用"
2. 填写应用基本信息（名称、描述）
3. 填写 SSH 连接信息（Host、用户名、端口，以及密码或私钥路径）
4. 填写服务端口（`service_port`，默认 8080）
5. 填写 JVM 进程名（`jvm_process_name`），用于在目标服务器上定位进程
6. 可选填写 `arex_app_id`（AREX 官方注册的应用标识，用于从 arex-storage 查询录制数据）
7. 设置采样率（`sample_rate`，0.0~1.0，1.0 表示全量录制）
8. 点击"保存"

**测试 SSH 连接：**

1. 在应用详情页点击"测试连接"按钮
2. 系统调用 `POST /api/v1/applications/{app_id}/test-connection`
3. 连接成功返回 `{"success": true}`，失败会返回具体错误信息

**挂载 Agent（完整流程）：**

1. 确保 SSH 连接测试通过
2. 点击"挂载 Agent"按钮
3. 系统异步执行以下步骤（后台任务）：
   - 将本地 `AR_AREX_AGENT_JAR_PATH` 路径的 JAR 文件通过 SFTP 上传到目标服务器 `/home/{ssh_user}/arex-agent/arex-agent.jar`
   - 在目标服务器的启动脚本（`start.sh` 或 `startup.sh`）中注入 `-javaagent:/home/{ssh_user}/arex-agent/arex-agent.jar=arex.service.name={app_name};arex.storage.service.host={arex_url}` 参数
   - 根据操作结果更新 `agent_status` 字段
4. Agent 状态显示为 `online` 表示挂载成功
5. 重启目标服务使 Agent 生效（平台不自动重启服务，需手动操作）

**查看 Agent 状态：**

点击"刷新状态"按钮调用 `GET /api/v1/applications/{app_id}/agent-status`，系统通过 SSH 登录目标服务器检测 JVM 进程是否包含 `-javaagent` 参数，实时更新状态。

**卸载 Agent：**

1. 点击"卸载 Agent"按钮
2. 系统通过 SSH 在启动脚本中删除 `-javaagent` 相关行（使用 `sed -i '/-javaagent/d'`）
3. `agent_status` 置为 `offline`
4. 重启目标服务使卸载生效

#### 字段说明

| 字段 | 说明 |
|------|------|
| `name` | 应用名称，建议与 arex-storage 中注册的应用名保持一致 |
| `ssh_host` | 目标服务器 IP 或域名 |
| `ssh_user` | SSH 登录用户名，该用户需有权限修改启动脚本 |
| `ssh_key_path` | SSH 私钥在本地的路径（`AR_SSH_KEYS_DIR` 目录下） |
| `ssh_password` | SSH 密码认证（与 `ssh_key_path` 二选一） |
| `ssh_port` | SSH 端口，默认 22 |
| `service_port` | 目标服务 HTTP 端口，回放时向此端口发送请求，默认 8080 |
| `jvm_process_name` | JVM 进程标识名，用于 `ps aux` 定位进程 |
| `arex_app_id` | AREX 应用标识，用于从 arex-storage 查询该应用的录制数据；若不填则使用应用名 |
| `arex_storage_url` | 覆盖全局 `AR_AREX_STORAGE_URL` 配置，用于该应用独立对接不同的 arex-storage 实例 |
| `sample_rate` | 录制采样率，0.0~1.0，生产环境建议设置 0.1~0.3 以控制录制数据量 |
| `agent_status` | Agent 当前状态：`unknown`（未知）/`online`（在线）/`offline`（离线）/`mounting`（挂载中） |
| `default_ignore_fields` | 该应用下所有用例的默认忽略字段列表（JSON 数组），可被用例级配置覆盖 |
| `default_assertions` | 该应用下所有用例的默认断言规则（JSON 数组） |
| `default_perf_threshold_ms` | 默认性能阈值（毫秒），超出此值的回放结果标记为性能异常 |
| `desensitize_rules` | 脱敏规则（JSON），配置哪些字段在存储前自动脱敏 |

#### 注意事项

- Agent 挂载操作为异步后台任务，点击按钮后状态会先显示 `mounting`，需等待几秒后刷新状态
- 挂载成功后必须**手动重启目标服务**，Agent 才会实际生效并开始录制
- `ssh_key_path` 中的私钥文件权限必须为 `600`，否则 Paramiko 会拒绝使用
- 如果目标服务器上没有 `start.sh` 或 `startup.sh`，注入操作会失败，此时需手动在启动命令中添加 `-javaagent` 参数

### 4.3 录制管理

#### 功能简介

录制管理模块负责从 arex-storage 同步录制数据，并以录制会话（Session）为单位组织管理。每个录制会话关联一个应用，包含若干录制条目（Recording），每条记录包含完整的 HTTP 请求、响应以及下游子调用信息。

#### 使用场景

- 在目标服务接收了一批真实流量后，将这批流量同步到平台作为测试用例的原材料
- 预览某次请求的完整调用链（包括 MySQL 子调用的入参和返回值）
- 将优质的录制条目批量转换为可复用的测试用例

#### 录制会话与录制条目的关系

```
RecordingSession（录制会话）
  └── Recording × N（录制条目）
        ├── 主请求（HTTP Method + URI + Headers + Body）
        ├── 主响应（Status + Headers + Body）
        └── sub_calls（JSON 数组：MySQL/Redis 等子调用的请求与响应）
```

一个录制会话对应一次同步操作，可以指定时间范围和分页参数，同一条录制（`record_id` 唯一）不会被重复导入。

#### 操作步骤

**创建录制会话并同步数据：**

1. 进入"录制管理" → 选择目标应用
2. 点击"新建会话"，输入会话名称（如"2026-04-10 早高峰流量"）
3. 点击"Sync 同步"按钮，配置同步参数：
   - **起始时间**（`begin_time`）：默认为当天零点
   - **结束时间**（`end_time`）：默认为当前时间
   - **每页条数**（`page_size`）：默认 20，最大 200
   - **页码**（`page_index`）：默认第 1 页
4. 确认后系统在后台向 arex-storage 发起查询，会话状态变为 `collecting`
5. 同步完成后状态变为 `done`，`total_count` 显示本次导入的新增条数

**查看录制条目：**

1. 点击会话名称进入条目列表
2. 每条录制显示：请求方法、URI、响应状态码、延迟时间
3. 点击某条录制查看详情：
   - **请求头**（`request_headers`）：JSON 格式
   - **请求体**（`request_body`）：原始请求内容
   - **响应体**（`response_body`）：录制时的原始响应
   - **子调用**（`sub_calls`）：JSON 数组，每个元素包含 `type`（如 `MySQL`）、`request`（SQL 语句）、`response`（查询结果）

**批量转为测试用例：**

1. 在录制列表中勾选目标条目（可多选）
2. 点击"转为用例"按钮
3. 系统调用 `POST /api/v1/test-cases/from-recording`，将录制的请求信息和响应体自动填入测试用例
4. 新建的用例默认状态为 `draft`，需进入测试用例库进一步编辑断言规则

**过滤录制条目：**

在会话录制列表中可通过以下方式过滤：
- **URI 过滤**：`uri_filter` 参数支持字符串包含匹配（调用 `GET /api/v1/sessions/recordings/all?uri_filter=/api/users`）
- **跨会话查看**：使用"全部录制"视图可跨会话按应用 ID 过滤

#### 字段说明

| 字段 | 说明 |
|------|------|
| `record_id` | AREX 原始录制 ID，全局唯一，防止重复导入 |
| `request_method` | HTTP 方法（GET/POST/PUT/DELETE 等） |
| `request_uri` | 请求路径，如 `/api/v1/users/123` |
| `request_headers` | 请求头 JSON，包含 Content-Type、Authorization 等 |
| `request_body` | 请求体，POST/PUT 请求的 JSON 或表单数据 |
| `response_status` | HTTP 响应状态码 |
| `response_body` | 录制时服务返回的响应体（作为后续对比的基准） |
| `sub_calls` | 子调用 JSON 数组，格式：`[{"type":"MySQL","request":"SELECT...","response":{...}}]` |
| `latency_ms` | 请求延迟（毫秒） |
| `tags` | 标签，逗号分隔字符串 |

#### 注意事项

- arex-storage 中的数据有保留期限，超期数据会被清理，请及时同步重要录制
- `sub_calls` 字段包含数据库查询结果，可能含有敏感数据，注意配置脱敏规则
- 同步时如果 `arex_app_id` 未配置，系统会使用应用的 `name` 字段作为查询标识，需确保与 arex-storage 中的应用名匹配

### 4.4 测试用例库

#### 功能简介

测试用例库是平台的核心资产。每个测试用例定义了一次 HTTP 请求的完整信息以及期望的验证规则，可以手动创建、从录制转化，或从 JSON 文件批量导入。用例通过状态和标签进行生命周期管理，支持跨应用共享和套件编排。

#### 使用场景

- 维护一套长期有效的回归测试用例，每次发版前执行验证
- 通过克隆用例快速生成边界条件变体（修改请求参数后测试异常路径）
- 导出用例 JSON 备份，或迁移到其他环境

#### 测试用例的组成

```
TestCase
├── 基本信息
│   ├── name          用例名称
│   ├── description   用例描述
│   ├── status        状态（draft/active/deprecated）
│   └── tags          标签（逗号分隔）
├── 请求定义
│   ├── request_method   HTTP 方法
│   ├── request_uri      请求路径（支持路径参数，如 /users/{id}）
│   ├── request_headers  请求头 JSON
│   └── request_body     请求体 JSON
├── 期望验证
│   ├── expected_status    期望 HTTP 状态码
│   ├── expected_response  期望响应体（从录制导入时自动填充）
│   ├── assert_rules       自定义断言规则 JSON 数组
│   └── ignore_fields      差异对比忽略字段 JSON 数组
└── 性能阈值
    └── perf_threshold_ms  响应时间上限（毫秒），超出视为性能失败
```

#### 状态管理

| 状态 | 含义 | 典型操作 |
|------|------|----------|
| `draft` | 草稿，从录制转化后的初始状态，尚未配置断言规则 | 编辑后激活 |
| `active` | 激活，可被套件和定时任务引用，参与常规回归测试 | 正常使用 |
| `deprecated` | 已废弃，接口已下线或需求已变更，保留历史记录 | 不参与执行 |

#### 操作步骤

**手动创建用例：**

1. 进入"测试用例" → 点击"新建用例"
2. 填写用例名称和描述
3. 选择关联应用（可选）
4. 填写请求方法和 URI
5. 在 Monaco Editor 中编辑请求头 JSON 和请求体 JSON
6. 填写期望状态码和期望响应体（可留空，仅使用断言验证）
7. 配置断言规则（见 4.6 节）
8. 配置忽略字段（JSONPath 列表）
9. 选择状态（建议先保存为 `draft`，验证后改为 `active`）
10. 点击"保存"

**克隆用例：**

1. 在用例列表找到目标用例
2. 点击"克隆"按钮，系统调用 `POST /api/v1/test-cases/{case_id}/clone`
3. 克隆后的用例名称自动添加"(copy)"后缀，状态重置为 `draft`
4. 进入克隆后的用例，修改请求参数或断言规则

**标签系统使用：**

- 标签以逗号分隔存储于 `tags` 字段，如 `"smoke,regression,user-api"`
- 在用例列表中通过 `tags` 查询参数过滤，如 `?tags=smoke` 会匹配包含"smoke"的所有用例
- 建议标签命名规范：`{模块}-{类型}`，如 `order-smoke`、`payment-regression`

**导入用例（JSON 格式）：**

导入文件为 JSON 数组，每个元素的字段对应 `TestCaseCreate` 结构：

```json
[
  {
    "name": "查询用户接口",
    "request_method": "GET",
    "request_uri": "/api/v1/users/1",
    "request_headers": "{\"Content-Type\": \"application/json\"}",
    "expected_status": 200,
    "expected_response": "{\"code\": 0, \"data\": {\"id\": 1}}",
    "assert_rules": "[{\"type\": \"json_path_eq\", \"path\": \"code\", \"value\": 0}]",
    "ignore_fields": "[\"timestamp\", \"traceId\"]",
    "tags": "user,smoke",
    "status": "active"
  }
]
```

导入步骤：
1. 进入"测试用例" → 点击"导入"
2. 选择 JSON 文件上传
3. 系统调用 `POST /api/v1/test-cases/import`，返回 `{"imported": N}`

**导出用例：**

```bash
# 导出全部用例
GET /api/v1/test-cases/export

# 按应用导出
GET /api/v1/test-cases/export?application_id=1

# 按 ID 列表导出
GET /api/v1/test-cases/export?ids=1,2,3,5
```

#### Monaco Editor 使用说明

请求头、请求体、期望响应、断言规则等 JSON 字段均通过 Monaco Editor 编辑：
- **格式化**：按 `Shift+Alt+F`（Windows/Linux）或 `Shift+Option+F`（macOS）自动格式化
- **校验**：编辑器会实时标出 JSON 语法错误
- **自动补全**：对已知 JSON 结构提供字段补全提示

#### 注意事项

- `request_headers` 和 `assert_rules` 等字段以字符串形式存储（数据库中为 Text 类型），JSON 序列化由前端负责
- 用例中 `ignore_fields` 是字段名数组（如 `["timestamp", "traceId"]`），系统会自动转为 deepdiff 的正则排除规则；`diff_rules` 则使用点分隔路径语法（见 4.6 节）
- 克隆操作会复制所有字段，包括断言规则和忽略字段，克隆后的用例 `source_recording_id` 保持与原始用例相同

### 4.5 回放执行

#### 功能简介

回放执行模块负责将测试用例列表作为输入，并发地向目标服务发送 HTTP 请求，将响应与期望值进行对比，生成回放任务（ReplayJob）和每条用例的执行结果（ReplayResult）。前端通过 WebSocket 实时接收进度推送。

#### 使用场景

- 新版本部署后，对核心接口进行回归验证
- 开发联调完成后，快速跑一遍全量测试用例确认无回归
- 通过套件执行，按业务模块分批验证

#### 操作步骤

**手动触发回放：**

1. 进入"测试用例"列表，勾选目标用例（或进入套件页面直接运行）
2. 点击"执行回放"
3. 配置回放参数：
   - **并发数**（`concurrency`）：1~50，建议测试环境 5~10，性能敏感场景设为 1
   - **超时时间**（`timeout_ms`）：单个请求超时时间（毫秒），默认 5000
   - **是否启用子调用 Mock**（`use_sub_invocation_mocks`）：开启后 AREX 自动 Mock MySQL 等子调用
   - **通知配置**（可选）：完成后推送到钉钉或企微 Webhook
4. 点击"开始执行"，系统创建 `ReplayJob` 并立即开始执行
5. 页面自动通过 WebSocket 连接到 `ws://localhost:8000/api/v1/replays/{job_id}/ws`，实时显示执行进度

**并发度配置建议：**

| 场景 | 建议并发数 | 原因 |
|------|------------|------|
| 单接口顺序调试 | 1 | 便于逐条分析结果 |
| 常规回归测试 | 5（默认） | 平衡速度与服务压力 |
| 大批量快速执行 | 10~20 | 测试环境资源充足时 |
| 生产环境谨慎执行 | 1~3 | 避免影响真实流量 |
| 最大并发 | 50 | 压力测试场景，谨慎使用 |

#### AREX 自动 Mock 的工作原理

当 `use_sub_invocation_mocks` 为 `true` 时：
- 回放请求发送到目标服务时，携带 AREX 特定的请求头（包含录制 ID）
- 目标服务中的 AREX Agent 识别到该请求头后，将 MySQL/Redis 等下游调用替换为返回录制时保存的子调用响应
- 这样即使数据库数据发生变化，也能保证回放结果可与原始录制响应进行稳定对比

此功能需要目标服务已挂载并运行 arex-agent，且录制条目中包含有效的 `sub_calls` 数据。

#### WebSocket 实时进度说明

连接建立后，服务端推送 JSON 格式的进度消息：

```json
{
  "job_id": 42,
  "done": 35,
  "total": 100,
  "passed": 30,
  "failed": 4,
  "errored": 1,
  "finished": false
}
```

当 `finished` 为 `true` 时，表示任务完成，前端关闭 WebSocket 连接。

#### 回放结果状态说明

| 状态 | 含义 | 对应 `is_pass` |
|------|------|----------------|
| `PASS` | 响应符合期望，所有断言通过，差异对比无异常 | `true` |
| `FAIL` | 断言失败或差异对比超出阈值 | `false` |
| `ERROR` | 连接失败、服务返回 5xx 等非预期错误 | `false` |
| `TIMEOUT` | 请求超过 `timeout_ms` 配置的超时时间无响应 | `false` |
| `PENDING` | 等待执行（初始状态） | `false` |

#### 字段说明

| 字段 | 说明 |
|------|------|
| `concurrency` | 并发执行线程数，范围 1~50 |
| `timeout_ms` | 单个请求超时时间（毫秒） |
| `total` / `passed` / `failed` / `errored` | 任务汇总计数 |
| `use_sub_invocation_mocks` | 是否启用 AREX 子调用 Mock |
| `diff_rules` | 差异对比规则（JSON），见 4.6 节 |
| `assertions` | 断言规则（JSON），见 4.6 节 |
| `webhook_url` | 完成后通知的 Webhook 地址 |
| `notify_type` | 通知类型：`dingtalk` / `wecom` |

#### 注意事项

- 并发数设置过高时，目标服务可能出现限流或连接池耗尽，建议先从低并发开始
- 回放请求发送到应用的 `service_port` 端口，请确认目标服务在执行回放时处于运行状态
- 任务一旦启动无法暂停，只能等待完成或等任务超时

### 4.6 差异对比

#### 功能简介

差异对比引擎基于 Python `deepdiff` 库，对录制时的原始响应（`expected_response`）和回放时的实际响应（`actual_response`）进行全字段递归对比。系统同时支持自定义忽略规则、智能降噪和多种断言类型，有效减少因动态字段引起的误报。

#### deepdiff 对比原理

系统调用 `DeepDiff(original, replayed, ignore_order=True, exclude_regex_paths=...)` 进行对比：
- **ignore_order=True**：JSON 数组顺序不同不视为差异
- 对比结果按变化类型分类：`values_changed`（值变更）、`dictionary_item_added`（新增字段）、`dictionary_item_removed`（删除字段）、`type_changes`（类型变化）等
- **差异分数（diff_score）**：`变更字段数 / 总字段数`，范围 0.0（完全相同）~1.0（完全不同）

#### 忽略字段规则

**方式一：用例级 `ignore_fields`**（字段名数组）

在测试用例的 `ignore_fields` 字段中配置字段名数组：

```json
["timestamp", "traceId", "requestId", "updatedAt"]
```

系统自动将每个字段名转为 deepdiff 的正则排除规则 `r".*\['{field}'\].*"`，匹配响应 JSON 中所有层级的同名字段。

**方式二：回放任务级 `diff_rules`**（路径规则数组）

在创建回放任务时，可以配置更精细的 `diff_rules`，支持四种规则类型：

| 规则类型 | 字段 | 说明 | 示例 |
|----------|------|------|------|
| `ignore` | `path` | 完全忽略指定路径的差异 | `{"type":"ignore","path":"data.timestamp"}` |
| `numeric_tolerance` | `path`, `tolerance` | 数值在容差范围内视为相同 | `{"type":"numeric_tolerance","path":"data.price","tolerance":0.01}` |
| `regex_match` | `path`, `pattern` | 字段值匹配正则则视为通过 | `{"type":"regex_match","path":"data.time","pattern":"\\d{4}-\\d{2}-\\d{2}"}` |
| `type_only` | `path` | 只校验类型，不校验具体值 | `{"type":"type_only","path":"data.id"}` |

**路径语法（点分隔）：**

```
data.timestamp          → 精确匹配 data 对象下的 timestamp 字段
items.*.price           → 匹配 items 数组中所有元素的 price 字段
result.list.0.name      → 匹配 result.list 数组第 0 个元素的 name 字段
```

**方式三：智能噪声过滤（smart_noise_reduction）**

在回放任务中启用 `smart_noise_reduction`，系统自动忽略 30+ 类常见动态字段，包括：

| 类别 | 字段示例 |
|------|----------|
| 时间字段 | `timestamp`, `createdAt`, `updatedAt`, `expireTime`, `date` |
| ID/序列号 | `requestId`, `traceId`, `spanId`, `serialNo`, `orderNo` |
| Token/Session | `token`, `sessionId`, `accessToken`, `refreshToken`, `jwt` |
| 随机值 | `random`, `nonce`, `signature`, `salt` |
| 地理坐标 | `lat`, `lng`, `longitude`, `latitude` |
| 版本号 | `version`, `build` |
| 统计值 | `count`, `totalCount`, `viewCount` |
| 毫秒时间戳 | `ms`, `Unix`, `epoch` |

#### 自定义断言类型详解

断言规则存储在 `assert_rules` 字段（JSON 数组），格式如下，每条规则包含 `type` 字段和规则特定参数：

**1. `status_code_eq`：检查 HTTP 状态码**

```json
{"type": "status_code_eq", "value": 200}
```

验证回放响应的 HTTP 状态码等于期望值。

**2. `response_not_empty`：检查响应非空**

```json
{"type": "response_not_empty"}
```

验证响应体不为空（不是 null、空字符串或纯空白）。

**3. `json_path_eq`：JSONPath 取值精确匹配**

```json
{"type": "json_path_eq", "path": "code", "value": 0}
{"type": "json_path_eq", "path": "data.user.id", "value": 123}
```

使用点分隔路径取值，与期望值做精确相等比较。数组索引：`items.0.name`。

**4. `json_path_contains`：包含检查**

```json
{"type": "json_path_contains", "path": "message", "value": "success"}
```

验证指定路径的字段值（转为字符串后）包含期望的子字符串。

**5. `json_path_exists`：字段存在性检查**

```json
{"type": "json_path_exists", "path": "data.id"}
```

验证指定路径的字段存在于响应 JSON 中（即使值为 `null` 也视为存在）。

**6. `json_path_regex`：正则匹配**

```json
{"type": "json_path_regex", "path": "data.createTime", "pattern": "\\d{4}-\\d{2}-\\d{2}"}
{"type": "json_path_regex", "path": "data.phone", "pattern": "^1[3-9]\\d{9}$"}
```

验证指定路径的字段值（转为字符串后）匹配正则表达式。

**7. `diff_score_lte`：差异分数阈值**

```json
{"type": "diff_score_lte", "value": 0.1}
```

验证 deepdiff 计算的差异分数不超过指定阈值（0.1 表示允许不超过 10% 的字段发生变化）。适用于允许少量动态字段变化但整体结构需保持一致的场景。

#### 综合示例

以下是一个完整的断言规则配置示例：

```json
[
  {"type": "status_code_eq", "value": 200},
  {"type": "response_not_empty"},
  {"type": "json_path_eq", "path": "code", "value": 0},
  {"type": "json_path_exists", "path": "data.userId"},
  {"type": "json_path_regex", "path": "data.email", "pattern": "^[\\w.-]+@[\\w.-]+\\.[a-z]{2,}$"},
  {"type": "diff_score_lte", "value": 0.05}
]
```

#### 注意事项

- `ignore_fields` 使用字段名进行全路径模糊匹配，适合忽略所有层级的同名字段；`diff_rules` 使用精确路径，适合精细控制
- `diff_score_lte` 的阈值应根据实际接口的动态字段比例来设置，通常 0.05~0.15 是合理范围
- 正则表达式需遵循 Python `re` 模块语法，特殊字符需要双重转义（JSON 中 `\\d` 表示正则的 `\d`）

### 4.7 结果分析

#### 功能简介

结果分析模块提供对回放任务执行结果的汇总查看、逐条分析和 HTML 报告导出功能。失败结果按失败类别分类，帮助快速定位问题根因。

#### 操作步骤

**查看回放报告：**

1. 进入"回放历史"，找到目标任务（状态为 DONE/FAILED）
2. 点击任务名称进入详情页，查看汇总信息：
   - 总用例数、通过数、失败数、错误数
   - 通过率（`passed / total * 100%`）
3. 在结果列表中筛选查看：
   - 点击"失败"标签，仅展示 `status=FAIL` 的结果
   - 点击"错误"标签，仅展示 `status=ERROR` 的结果
4. 点击某条失败结果查看详情：
   - **实际响应** vs **期望响应**：并排展示，便于对比
   - **差异详情**（`diff_result`）：deepdiff 输出的 JSON，列出每个差异字段
   - **断言结果**（`assertion_results`）：每条断言的通过/失败状态和详细消息

**失败类型分类说明：**

| `failure_category` | 含义 | 排查方向 |
|--------------------|------|----------|
| `diff` | 响应体字段值与期望不符 | 检查是否有动态字段未忽略，或接口逻辑真正发生变化 |
| `assertion` | 自定义断言规则未通过 | 检查断言配置是否合理，或接口返回码/结构变化 |
| `timeout` | 请求超过超时时间无响应 | 检查目标服务性能、网络连通性，或增大 `timeout_ms` |
| `connection_error` | 无法建立 HTTP 连接 | 检查目标服务是否运行、`service_port` 是否正确、网络策略 |

**导出 HTML 报告：**

```bash
# 浏览器直接访问（需 Bearer Token，或通过前端页面点击"导出报告"）
GET /api/v1/replays/{job_id}/report
```

HTML 报告为独立的单文件，包含：
- 任务概要（状态、通过率、总数统计）
- 结果明细表格（每条用例的请求、状态、HTTP 状态码、延迟、失败原因）
- 通过绿色/红色颜色区分通过和失败

报告可以直接分享给开发同学或附件到 Issue 中，无需平台账号即可查看。

#### 注意事项

- 结果列表默认每页 50 条，最多 200 条；超大任务需分页查看
- `diff_result` 字段存储完整的 deepdiff JSON 输出，可能较长，前端做了折叠展示
- HTML 报告生成为实时接口，数据来自数据库，任务完成后任意时间均可重复导出

### 4.8 测试套件

#### 功能简介

测试套件（Suite）是测试用例的有序集合，一个套件可以包含来自不同应用的多个测试用例，并定义用例的执行顺序。套件是定时任务和 CI/CD 集成的核心执行单元。

#### 使用场景

- 将核心冒烟测试用例归集为一个"冒烟套件"，每次部署后快速执行
- 按业务模块组织套件（如"订单模块回归"、"用户模块回归"）
- 通过定时任务每天自动执行套件，实现持续质量监控

#### 套件与用例的关系

套件和用例之间是**多对多**关系，通过 `replay_suite_case` 关联表维护：

```
replay_suite（套件）
  └── replay_suite_case × N（关联表，含 order_index 排序字段）
        └── test_case（测试用例）
```

同一个用例可以被加入多个套件，同一个套件中的用例可以通过 `order_index` 字段控制执行顺序。

#### 操作步骤

**创建套件：**

1. 进入"测试套件" → 点击"新建套件"
2. 输入套件名称和描述
3. 点击"保存"，创建空套件

**向套件添加用例（方式一：从用例库添加）：**

1. 进入测试用例列表，找到目标用例
2. 点击"加入套件"
3. 在弹窗中选择目标套件
4. 系统调用 `POST /api/v1/test-cases/{case_id}/add-to-suite`，自动分配 `order_index`（当前最大值+1）

**向套件添加用例（方式二：套件管理页面批量设置）：**

1. 进入套件详情页
2. 点击"设置用例"，在多选框中选择全部期望加入的用例
3. 确认后系统调用 `PUT /api/v1/suites/{suite_id}/cases`，传入有序的 `case_ids` 数组，全量替换套件内用例

**调整用例执行顺序：**

1. 在套件详情页，拖拽用例卡片调整顺序
2. 系统调用 `POST /api/v1/suites/{suite_id}/reorder`，更新每个用例的 `order_index`
3. 回放执行时，系统按 `order_index` 升序的顺序读取用例

**立即执行套件：**

1. 在套件详情页点击"执行"按钮
2. 系统调用 `POST /api/v1/suites/{suite_id}/run`，自动创建 `ReplayJob`（名称为 `Suite Run: {suite.name}`，默认并发 5，超时 5000ms）
3. 返回 `job_id`，可跳转到回放历史页查看进度

#### 字段说明

| 字段 | 说明 |
|------|------|
| `name` | 套件名称 |
| `description` | 套件描述 |
| `order_index` | 用例在套件中的执行顺序，从 0 开始，数字越小越先执行 |

#### 注意事项

- 删除套件时，关联表 `replay_suite_case` 中的记录会级联删除，但测试用例本身不会删除
- 一个用例重复加入同一个套件时，系统会返回 `{"message": "Already in suite"}` 而非报错
- 套件执行时并发度和超时时间使用系统默认值（5 和 5000ms），如需自定义请通过 CI/CD 接口触发

### 4.9 定时任务

#### 功能简介

定时任务模块基于 APScheduler 实现，支持通过 Cron 表达式配置定期自动执行测试套件，并在执行完成后通过钉钉或企业微信推送通知。系统启动时会自动从数据库加载所有激活状态的定时任务。

#### 使用场景

- 每天早上 9 点自动执行冒烟测试，及时发现夜间变更引入的问题
- 每次 CI 部署完成后（结合 Webhook）自动触发回归测试
- 每周五定期执行全量回归，生成周报并推送给团队

#### Cron 表达式格式

系统使用标准 5 字段 Cron 格式（与 Linux crontab 相同）：

```
分钟(0-59)  小时(0-23)  日(1-31)  月(1-12)  星期(0-7, 0和7都表示周日)
```

**常用示例：**

| Cron 表达式 | 含义 |
|-------------|------|
| `0 9 * * 1-5` | 工作日（周一到周五）早上 9:00 |
| `0 */2 * * *` | 每 2 小时执行一次 |
| `0 0 * * *` | 每天午夜 0:00 |
| `30 8 * * 1` | 每周一早上 8:30 |
| `0 18 * * 5` | 每周五下午 18:00 |
| `*/15 * * * *` | 每 15 分钟执行一次 |
| `0 9,12,18 * * 1-5` | 工作日 9:00、12:00、18:00 各执行一次 |

#### 操作步骤

**创建定时任务：**

1. 进入"定时任务" → 点击"新建任务"
2. 填写任务名称
3. 选择关联的测试套件（`suite_id`）
4. 填写 Cron 表达式（`cron_expr`），例如 `0 9 * * 1-5`
5. 配置通知：
   - **通知类型**（`notify_type`）：选择 `dingtalk`（钉钉）或 `wecom`（企业微信）或不通知
   - **Webhook 地址**（`notify_webhook`）：填写机器人 Webhook URL
6. 可选配置：
   - `override_host`：覆盖应用的默认 Host，用于指向特定环境
   - `diff_rules` 和 `assertions`：本次定时任务专用的对比规则
   - `perf_threshold_ms`：性能阈值
7. 确认 `is_active` 为 `true`（激活状态）
8. 点击"保存"，任务立即注册到 APScheduler

**获取钉钉/企微 Webhook 地址：**

- **钉钉**：打开钉钉群 → 群设置 → 智能群助手 → 添加机器人 → 自定义机器人 → 复制 Webhook URL
- **企业微信**：打开企微群 → 右键/更多 → 添加群机器人 → 新建机器人 → 复制 Webhook 地址

**通知消息内容：**

执行完成后推送的通知消息（Markdown 格式）包含：
```
AREX Recorder 回放完成 ✅/❌
- 回放任务: Suite Run: 冒烟测试套件
- 状态: ✅ 全部通过 / ❌ 存在失败
- 通过率: 95.0%
- 总数: 100 | 通过: 95 | 失败: 5 | 错误: 0
- 报告链接: http://localhost:8000/api/v1/replays/42/report
```

**立即触发定时任务（手动）：**

1. 在定时任务列表找到目标任务
2. 点击"立即触发"按钮
3. 系统调用 `POST /api/v1/schedules/{schedule_id}/trigger`，不等待下次 Cron 时间直接执行

**查看执行历史：**

定时任务表中的 `last_run_at` 和 `last_run_status` 字段记录最近一次执行时间和状态，完整的执行历史记录在 `replay_job` 表中（任务名称格式为 `Suite Run: {suite_name}`）。

#### 字段说明

| 字段 | 说明 |
|------|------|
| `name` | 任务名称 |
| `suite_id` | 关联的测试套件 ID |
| `cron_expr` | Cron 表达式，5 字段格式 |
| `is_active` | 是否激活，`false` 时任务不会触发 |
| `notify_type` | 通知类型：`dingtalk` / `wecom` / `none` |
| `notify_webhook` | 通知 Webhook URL |
| `last_run_at` | 最近一次触发时间 |
| `last_run_status` | 最近一次执行状态 |
| `override_host` | 覆盖请求目标 Host，如 `http://staging.example.com:8080` |
| `diff_rules` | 本任务专用的差异对比规则（JSON） |
| `perf_threshold_ms` | 性能阈值（毫秒） |

#### 注意事项

- 修改定时任务后，系统会先注销旧的调度器任务再注册新的，更新立即生效
- 定时任务的时区与服务器系统时区一致，部署时需注意服务器时区配置
- 如果 APScheduler 在任务触发时正在处理其他任务，新任务会在队列中等待，不会丢失
- Webhook 地址需保持可访问状态，推送失败时系统记录日志但不重试

### 4.10 CI/CD 集成

#### 功能简介

CI/CD 集成模块提供独立于用户 JWT 认证的 API Token 机制，允许 Jenkins、GitHub Actions 等 CI 系统通过 HTTP 接口触发套件回放并轮询结果，实现自动化质量门禁。

#### 使用场景

- 在 GitHub Actions workflow 中，部署完成后自动触发回放，通过率低于阈值时阻断合并
- Jenkins pipeline 集成，每次构建后执行冒烟测试套件，失败则标记构建不稳定
- 跨系统通知：Webhook 回调将回放结果推送到 Jira、Slack 等工具

#### API Token 创建与管理

Token 管理仅限 `admin` 角色：

1. 进入"CI/CD 集成" → "Token 管理"
2. 点击"创建 Token"
3. 填写：
   - **名称**（`name`）：描述用途，如"Jenkins-冒烟测试"
   - **权限范围**（`scope`）：`trigger`（可触发回放）或 `read_only`（仅查询结果）
   - **有效期**（`expires_days`）：可选，不填则永久有效
4. 点击"创建"，系统生成 Token 并**仅在此刻显示明文**（`plain_token`字段），复制保存后无法再次查看
5. Token 以 bcrypt 哈希存储于数据库，平台本身无法恢复明文

**吊销 Token：**

在 Token 列表中点击"吊销"，调用 `DELETE /api/v1/ci/tokens/{token_id}`，`is_active` 置为 `false`，立即失效。

#### 触发回放 API

**请求格式：**

```bash
curl -X POST http://localhost:8000/api/v1/ci/trigger \
  -H "Authorization: Token your-plain-token-here" \
  -H "Content-Type: application/json" \
  -d '{
    "suite_id": 3,
    "concurrency": 5,
    "timeout_ms": 5000,
    "notify_type": "dingtalk",
    "notify_webhook": "https://oapi.dingtalk.com/robot/send?access_token=xxx"
  }'
```

**响应：**

```json
{
  "job_id": 42,
  "status": "PENDING",
  "total": 100
}
```

#### 查询结果 API

```bash
curl http://localhost:8000/api/v1/ci/result/42 \
  -H "Authorization: Token your-plain-token-here"
```

**响应：**

```json
{
  "job_id": 42,
  "status": "DONE",
  "total": 100,
  "passed": 95,
  "failed": 5,
  "errored": 0,
  "pass_rate": 0.95
}
```

`status` 为 `PENDING` 或 `RUNNING` 时表示尚未完成，需继续轮询。

#### Jenkins Pipeline 集成示例

```groovy
pipeline {
    agent any
    stages {
        stage('Deploy') {
            steps {
                sh 'kubectl apply -f k8s/deployment.yaml'
                sh 'sleep 30'  // 等待服务启动
            }
        }
        stage('Smoke Test') {
            steps {
                script {
                    // 触发回放
                    def resp = sh(
                        script: '''curl -s -X POST http://arex-recorder:8000/api/v1/ci/trigger \
                            -H "Authorization: Token ${AREX_CI_TOKEN}" \
                            -H "Content-Type: application/json" \
                            -d '{"suite_id": 3, "concurrency": 5, "timeout_ms": 5000}' ''',
                        returnStdout: true
                    ).trim()
                    def jobId = readJSON(text: resp).job_id

                    // 轮询结果（最多等待 10 分钟）
                    def maxRetries = 60
                    def result = null
                    for (int i = 0; i < maxRetries; i++) {
                        sleep 10
                        def pollResp = sh(
                            script: "curl -s http://arex-recorder:8000/api/v1/ci/result/${jobId} -H 'Authorization: Token ${AREX_CI_TOKEN}'",
                            returnStdout: true
                        ).trim()
                        result = readJSON(text: pollResp)
                        if (result.status == 'DONE' || result.status == 'FAILED') break
                    }

                    // 质量门禁：通过率低于 95% 时失败
                    if (result.pass_rate < 0.95) {
                        error("Smoke test failed! Pass rate: ${result.pass_rate * 100}%")
                    }
                    echo "Smoke test passed! Pass rate: ${result.pass_rate * 100}%"
                }
            }
        }
    }
}
```

在 Jenkins 中配置 Credential：`AREX_CI_TOKEN` = 平台生成的 Token 明文。

#### GitHub Actions 集成示例

```yaml
name: Deploy and Smoke Test

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to staging
        run: kubectl apply -f k8s/

      - name: Wait for service ready
        run: sleep 30

      - name: Trigger AREX smoke test
        id: trigger
        run: |
          RESPONSE=$(curl -s -X POST ${{ vars.AREX_URL }}/api/v1/ci/trigger \
            -H "Authorization: Token ${{ secrets.AREX_CI_TOKEN }}" \
            -H "Content-Type: application/json" \
            -d '{"suite_id": 3, "concurrency": 5, "timeout_ms": 5000}')
          JOB_ID=$(echo $RESPONSE | jq -r '.job_id')
          echo "job_id=$JOB_ID" >> $GITHUB_OUTPUT

      - name: Poll test result
        run: |
          JOB_ID=${{ steps.trigger.outputs.job_id }}
          for i in $(seq 1 60); do
            RESULT=$(curl -s ${{ vars.AREX_URL }}/api/v1/ci/result/$JOB_ID \
              -H "Authorization: Token ${{ secrets.AREX_CI_TOKEN }}")
            STATUS=$(echo $RESULT | jq -r '.status')
            if [ "$STATUS" = "DONE" ] || [ "$STATUS" = "FAILED" ]; then
              PASS_RATE=$(echo $RESULT | jq -r '.pass_rate')
              echo "Test completed. Pass rate: $PASS_RATE"
              # 通过率低于 0.95 时退出码非 0，阻断流水线
              python3 -c "import sys; sys.exit(0 if float('$PASS_RATE') >= 0.95 else 1)"
              break
            fi
            sleep 10
          done
```

#### Webhook 回调配置

在触发回放时配置 `notify_webhook` 参数，回放完成后系统向该地址推送 Markdown 格式通知，可接入企业通知聚合系统。

#### 注意事项

- CI Token 认证格式为 `Authorization: Token <plain_token>`，注意与 JWT 认证的 `Bearer` 前缀不同
- Token 明文只在创建时返回一次，请立即复制到 CI 系统的 Secret 存储中
- 轮询结果时建议间隔 10 秒，避免频繁请求；若超过预期时间仍未完成，可查看 `status` 是否为 `RUNNING`（表示仍在执行）还是 `PENDING`（未开始，可能服务异常）

### 4.11 Dashboard 统计

#### 功能简介

Dashboard 提供系统全局和分应用维度的统计数据，帮助团队了解整体测试质量趋势，快速识别质量下滑的应用或时段。

#### 统计卡片含义

| 统计项 | API | 说明 |
|--------|-----|------|
| **注册应用数** | `GET /api/v1/stats/summary` → `apps` | 系统中注册的应用总数 |
| **测试用例总数** | `summary` → `test_cases` | 所有状态（含 draft/deprecated）的用例总数 |
| **近30天回放任务数** | `summary` → `recent_jobs` | 最近 30 天内状态为 DONE 的回放任务数 |
| **平均通过率** | `summary` → `avg_pass_rate` | 近 30 天所有 DONE 任务的加权平均通过率 |

#### 通过率趋势图

- 接口：`GET /api/v1/stats/trend?days=7&application_id={id}`
- 返回最近 N 天（默认 7 天，最长 90 天）的每日汇总
- 响应格式：
  ```json
  [
    {"date": "2026-04-01", "total": 200, "passed": 185, "pass_rate": 0.925},
    {"date": "2026-04-02", "total": 0, "passed": 0, "pass_rate": null}
  ]
  ```
- `pass_rate` 为 `null` 表示当天没有已完成的回放任务
- 前端使用 ECharts 折线图展示，可通过下拉框选择查看全局或某个应用的趋势

#### 失败分布

- 接口：`GET /api/v1/stats/failure-types?days=7`
- 返回最近 N 天失败结果按 `failure_category` 分组的计数
- 响应格式：
  ```json
  [
    {"category": "diff", "count": 45},
    {"category": "assertion", "count": 12},
    {"category": "timeout", "count": 3},
    {"category": "connection_error", "count": 1}
  ]
  ```
- 前端使用 ECharts 饼图展示各类失败的占比

#### 分应用统计

- 接口：`GET /api/v1/stats/app-summary`
- 展示每个应用的：测试用例数、近期任务数、近30天通过率、Agent 状态
- 帮助识别哪个应用的质量趋势在下降，哪个应用的 Agent 处于离线状态

#### 近期任务列表

- 接口：`GET /api/v1/stats/recent-jobs?limit=10`
- 返回最近 10 个有效状态（DONE/FAILED/RUNNING）的回放任务
- 包含每个任务的通过率，方便在 Dashboard 上快速看到最新执行结果

### 4.12 系统设置

#### 功能简介

系统设置模块管理平台级别的全局配置，包括 AREX storage 连接配置、默认回放参数和数据脱敏规则。这些配置通过环境变量在启动时加载，系统设置页面提供运行时查看和部分参数的动态调整能力。

#### AREX Storage URL 配置

全局 AREX storage 地址由 `AR_AREX_STORAGE_URL` 环境变量控制。如果某个应用需要独立对接不同的 arex-storage 实例（多租户场景），可以在应用设置的 `arex_storage_url` 字段进行覆盖。优先级：应用级配置 > 全局配置。

#### 全局超时与并发设置

| 参数 | 环境变量 | 说明 |
|------|----------|------|
| 默认回放并发数 | `AR_DEFAULT_REPLAY_CONCURRENCY` | 新建回放任务时的并发数初始值（可在任务创建时覆盖） |
| 默认回放超时 | `AR_DEFAULT_REPLAY_TIMEOUT_MS` | 单个请求超时时间初始值（毫秒，可覆盖） |

#### 脱敏规则配置

脱敏规则配置在应用级别的 `desensitize_rules` 字段（JSON 格式），定义哪些字段在录制数据同步时自动替换为脱敏值：

```json
[
  {"field": "password", "strategy": "mask"},
  {"field": "phone", "strategy": "partial", "keep_start": 3, "keep_end": 4},
  {"field": "idCard", "strategy": "hash"},
  {"field": "creditCard", "strategy": "mask"}
]
```

常见脱敏策略：
- `mask`：整体替换为 `***`
- `partial`：保留首尾 N 位，中间替换为 `****`
- `hash`：替换为 SHA256 哈希值（保留格式，便于比对）

脱敏规则在录制数据同步时由 `backend/utils/desensitize` 模块应用，确保敏感数据不以明文存储于本平台数据库。

### 错误码说明

| HTTP 状态码 | 含义 | 常见原因 |
|-------------|------|----------|
| 400 | 请求参数错误 | 必填字段缺失、格式不正确（如 Cron 表达式非法）、业务规则冲突（如 case_ids 为空） |
| 401 | 未认证 | Token 未提供、格式错误、Token 已过期、CI Token 已吊销 |
| 403 | 权限不足 | 当前角色无权执行此操作（如 Viewer 尝试创建用例） |
| 404 | 资源不存在 | 指定的 app_id/case_id/job_id 等不存在 |
| 422 | 数据验证失败 | Pydantic 参数验证失败，响应体中包含 `detail` 数组说明具体字段错误 |
| 500 | 服务器内部错误 | 数据库异常、SSH 连接异常、外部服务不可达等；开启 `AR_DEBUG=true` 可查看堆栈 |

## 8. 常见问题

**Q: Agent 挂载成功后录制数据为何没有出现？**

A: 挂载 Agent 并不会立即开始录制，需要完成以下步骤：
1. 确认挂载操作完成（`agent_status` 显示为 `online`）
2. **重启目标 Java 服务**，使 `-javaagent` 参数生效（平台不自动重启服务）
3. 向目标服务发送实际 HTTP 请求，触发录制
4. 等待几秒后，到录制管理页创建录制会话并点击"Sync"同步数据
5. 如仍无数据，检查 arex-storage 服务是否正常运行，以及 `arex_app_id` 是否与 arex-storage 中注册的应用名称一致

---

**Q: 回放结果都是 TIMEOUT，怎么排查？**

A: 按以下步骤排查：
1. 确认目标服务正在运行且 `service_port` 配置正确
2. 在平台服务器上手动 `curl http://{ssh_host}:{service_port}/actuator/health` 验证网络连通性
3. 检查 `timeout_ms` 设置是否过小（如设置了 1000ms 而接口正常响应需要 2000ms）
4. 如果并发数过高，目标服务可能过载导致响应变慢，尝试将并发数降至 1
5. 查看目标服务日志，确认是否收到请求

---

**Q: 差异对比有很多噪声字段，如何配置忽略规则？**

A: 有三种方式，从简到详：

**方式一（推荐入门）**：在回放任务中开启"智能降噪"（`smart_noise_reduction`），自动忽略 30+ 常见动态字段（timestamp、traceId 等）。

**方式二（用例级）**：在测试用例的 `ignore_fields` 字段填写字段名数组：
```json
["updateTime", "requestId", "sign", "nonce"]
```

**方式三（精细控制）**：在回放任务或定时任务的 `diff_rules` 字段配置路径规则：
```json
[
  {"type": "ignore", "path": "data.updateTime"},
  {"type": "ignore", "path": "result.*.createAt"},
  {"type": "numeric_tolerance", "path": "data.price", "tolerance": 0.001}
]
```

建议先用智能降噪，再根据实际误报情况逐步补充精细规则。

---

**Q: 定时任务没有触发通知，如何排查？**

A: 按以下步骤排查：
1. 确认 `notify_type` 已设置为 `dingtalk` 或 `wecom`，且 `notify_webhook` 地址填写正确
2. 在平台服务器上测试 Webhook 连通性：`curl -X POST <webhook_url> -H "Content-Type: application/json" -d '{"msgtype":"text","text":{"content":"test"}}'`
3. 查看后端日志是否有 `DingTalk send failed` 或 `WeCom notify error` 的警告信息
4. 确认钉钉机器人安全设置：若配置了"加签"或"IP 白名单"，确保平台服务器 IP 在白名单内
5. 确认任务确实触发了回放（查看 `replay_job` 表中是否有对应时间点的任务记录）

**Q: 如何重置管理员密码？**

A: 有两种方式：

**方式一（推荐）**：通过 API 以其他管理员账号更新密码：
```bash
curl -X PUT http://localhost:8000/api/v1/users/{user_id} \
  -H "Authorization: Bearer <other_admin_token>" \
  -H "Content-Type: application/json" \
  -d '{"password": "NewPassword123"}'
```

**方式二（直接操作数据库）**：
```python
# 生成新密码哈希（在 Python 环境中执行）
from passlib.context import CryptContext
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")
print(pwd_context.hash("NewPassword123"))
```

```sql
-- 更新数据库中的密码哈希
UPDATE user SET hashed_password = '$2b$12$...' WHERE username = 'admin';
```

**Q: 前端报 401，但 token 还有有效期？**

A: 可能原因：

1. **`AR_SECRET_KEY` 发生变更**：修改 JWT 密钥后，所有已颁发的 Token 立即失效，需重新登录
2. **服务重启导致内存状态丢失**：JWT 是无状态的，只要密钥不变，重启不影响 Token 有效性；如果重启后 Token 失效，请检查是否 `AR_SECRET_KEY` 使用了随机生成的临时值（每次启动不同）
3. **Token 携带格式错误**：确认请求头格式为 `Authorization: Bearer <token>`，`Bearer` 和 Token 之间有一个空格
4. **用户账号被停用**：管理员将用户的 `is_active` 置为 `false` 后，即使 Token 未过期，API 也会返回 401
5. **时钟偏差**：服务器时间与颁发 Token 的服务器时间相差过大，导致 JWT 认为 Token 已过期；检查服务器 NTP 同步状态
