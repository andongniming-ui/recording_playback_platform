### 1.2 子调用（Sub-call）录制 — ✅ 完全支持

### 1.3 数据库调用录制 — ✅ 完全支持

数据库调用录制通过 **AREX Agent 的 DynamicClass 机制**实现：

**录制流程：**
1. AREX Agent 拦截配置的 Repository 方法调用
2. 将方法名、参数、返回值打包为 mocker
3. 通过 `batchSaveMockers` 上报到 arex-storage（或本地 arex_proxy）
4. 平台从 arex_mocker 表读取并归一化

**归一化逻辑：**
- `normalize_repository_sub_call()`：有 Repository 映射的方法（如 `CarDataRepository.findVehicle`）
- `normalize_generic_database_sub_call()`：无映射的原生 JDBC/MySQL 调用

**关键代码路径：**
- `repository_capture.py` `normalize_repository_sub_call()` (第 477-506 行)
- `repository_capture.py` `normalize_generic_database_sub_call()` (第 325-385 行)
- `arex_proxy.py` `batch_save_mockers()` (第 391-454 行)：接收 Agent 上报

## 二、比对能力总览

### 2.1 HTTP 响应比对 — ✅ 完全支持

| 比对维度 | 实现方式 | 说明 |
|----------|----------|------|
| 状态码比对 | 直接比较 | `case.expected_status == actual_status` |
| 响应体比对 | DeepDiff | 支持 JSON/XML 结构化比对 |
| 智能降噪 | 30+ 内置规则 | 自动忽略时间戳、ID、Token 等动态字段 |
| 自定义忽略字段 | `ignore_fields` | 用户指定字段名列表 |
| 差异规则 | `diff_rules` | ignore/numeric_tolerance/regex_match/type_only |
| 差异分数 | 0.0-1.0 | `changed_keys / total_keys` |

**关键代码路径：**
- `diff.py` `compute_diff()` (第 57-122 行)
- `diff.py` `BUILTIN_SMART_NOISE_PATTERNS` (第 21-46 行)：30+ 内置降噪规则
- `replay_executor.py` `_execute_single()` (第 1075-1532 行)：主响应比对

---

### 2.2 子调用比对 — ✅ 完全支持

**比对流程：**
1. **配对**：按 `type + operation` 贪心匹配录制侧和回放侧子调用
2. **数量检查**：数量不一致直接判定为有差异
3. **响应内容比对**：
   - 启用智能比对时：使用 `compute_diff()`（支持降噪和自定义规则）
   - 未启用时：使用 `_deep_equal()` 深度相等检查
4. **严格模式**：`fail_on_sub_call_diff=True` 时，子调用差异会导致用例 FAIL

**关键代码路径：**
- `replay_executor.py` `_sub_calls_have_diff()` (第 91-177 行)
- `replay_executor.py` `_build_sub_call_diff_pairs()` (第 180-296 行)
- `replay_executor.py` `_strict_sub_call_failure()` (第 339-362 行)

**子调用差异结果结构：**
```json
{
  "type": "MySQL",
  "operation": "SELECT customer",
  "matched": true,
  "has_diff": true,
  "expected_response": {...},
  "actual_response": {...},
  "diff_result": "{\"values_changed\": {...}}"
}
```

### 2.3 数据库调用比对 — ✅ 完全支持

数据库调用作为子调用的一种类型，比对逻辑与其他子调用一致，但有以下特殊处理：

| 特殊处理 | 说明 |

| SQL 文本提取 | 从 request attributes 或 request body 中提取 SQL |

| 前端展示 | SQL/参数单独卡片展示 |

**关键代码路径：**
- `repository_capture.py` `normalize_generic_database_sub_call()` (第 325-385 行)
- `sessions.py` `_database_sub_call_identity()` (第 570-580 行)

## 三、支持的子调用类型

### 3.1 AREX Agent 原生支持的 categoryType

| categoryType | 说明 | 录制方式 |
|-------------|------|----------|
| `Servlet` | HTTP 入口请求 | Agent 自动拦截 |
| `HttpClient` | HTTP 外部调用（RestTemplate 等） | Agent 自动拦截 |
| `DynamicClass` | 动态类方法调用 | 需配置 `dynamicClassConfigurationList` |
| `MySQL` | MySQL 数据库调用 | Agent 自动拦截（需 JDBC 驱动支持） |
| `JDBC` | 通用 JDBC 调用 | Agent 自动拦截 |

## 五、已知限制与注意事项

| 限制项 | 影响 | 缓解措施 |
|--------|------|----------|
| Agent flush 延迟 | 回放子调用可能为空 | 默认等待 1s + 3 次重试，可配置 `AR_AREX_FLUSH_DELAY_S` |
| 多协程计数竞争 | `progress_lock` 在单进程有效，多进程部署时可能不准确 | 使用数据库原子操作 `func.coalesce()` |
| ArexClient 连接池 | 每次请求新建连接 | 当前实现未复用连接池，高频场景可优化 |
| 子调用配对策略 | 贪心匹配可能不是最优配对 | 当前按 `type + operation` 匹配，复杂场景可优化 |
| 数据库调用依赖配置 | JdbcTemplate/原生 JDBC 需 DynamicClass 配置 | 已在 `repository_capture.py` 中配置 |
| 内部 Servlet 过滤 | Agent 会录制内部 HTTP 调用为独立 entry-point | `_remove_sub_invocation_recordings()` 自动过滤 |

## 六、结论

### 录制能力

| 能力 | 状态 |
|------|------|
| HTTP 请求/响应录制 | ✅ 完全支持 |
| 子调用录制 | ✅ 完全支持（多源融合） |
| 数据库调用录制 | ✅ 完全支持（DynamicClass 机制） |
| HTTP 外呼录制 | ✅ 完全支持（HttpClient category） |
| 缓存调用录制 | ✅ 支持（Redis category） |
| RPC 调用录制 | ✅ 支持（Dubbo/gRPC category） |
| 消息队列录制 | ✅ 支持（Kafka/RabbitMQ category） |

### 比对能力

| 能力 | 状态 |
|------|------|
| HTTP 响应比对 | ✅ 完全支持（DeepDiff + 智能降噪） |
| 子调用比对 | ✅ 完全支持（配对 + 响应内容比对） |
| 数据库调用比对 | ✅ 完全支持（作为子调用子集） |
| 严格模式 | ✅ 支持（`fail_on_sub_call_diff`） |
| 差异详情展示 | ✅ 完全支持（前后端协同） |

**总体结论：该平台具备完整的录制和比对能力，支持 HTTP 请求/响应、子调用（含数据库调用、外呼、缓存、RPC、消息队列）的全链路录制与精细比对。**
