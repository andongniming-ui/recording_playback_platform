一、核心结论

1. 当前平台已经具备完整主链路

对 `didi` 样例系统，平台已实现“真实请求录制 -> 原始数据落库 -> 录制同步 -> 测试用例沉淀 -> 回放比对”的完整闭环， 不是停留在页面展示或单点录包。

2. 当前最适合的目标系统是 Java 服务

平台最适配并已验证的是 Java / Spring Boot 服务，尤其适合带 HTTP 接口、内部 HTTP 子调用、MySQL/JDBC 访问的系统； 对多环境同接口比对尤其有效。

3. 已抓到的数据可验证为真实链路数据

现有数据可以从业务日志、平台原始表、平台录制表、测试用例、回放结果以及业务数据库 6 个层面做交叉验证， 可追溯到同一笔交易。

三、可以覆盖的场景

平台当前最适合解决的是“真实请求驱动的回归验证”和“多环境差异定位”。

版本回归

把历史真实请求沉淀成用例，在新版本或新环境上回放，快速发现功能回归、返回值变化和性能波动。

双环境比对

适合“老系统 vs 新系统”“基线环境 vs 对比环境”的一致性验证，尤其适合接口相同但内部实现不同的场景。

复杂链路追踪

不只看入口返回，还能看到内部 HTTP 子调用和数据库操作，适合分析复杂交易为何出现差异。

定时回归 / 持续验证

适合每日回归、发版前冒烟、接口稳定性巡检，以及接入流水线后的自动触发验证。

生产流量沉淀为测试资产 接口升级前后对比 疑难问题复现 关键交易长期守护 多团队共享回归样本

五、测试外部系统时，需要对方提供什么

分成“必需项”和“建议项”。必需项缺失时，平台很难形成闭环；建议项会直接影响验证效率和可信度。

必需项

  • 被测服务的访问地址、端口、环境信息。
  • 服务启动方式可控：能修改 `start.sh`、启动命令或 Docker Compose 参数，以便挂载 `-javaagent`。
  • 服务所在机器的 SSH 信息或容器工作目录信息，用于平台纳管。
  • 应用标识、进程名、服务端口，便于平台识别并同步录制数据。
  • 基础调用样本：接口文档、示例请求、测试账号、必要的业务测试数据。
  • 网络连通：业务系统所在机器能够访问录制平台。

建议项

  • 日志中保留统一业务标识，如 `traceId`、`traId`、`request_no`、交易码。
  • 提供业务数据库只读账号，便于核对落库结果是否与录制一致。
  • 明确关键交易清单和高优先级场景，先从高价值交易录制。
  • 准备一套稳定的基线环境和一套对比环境，便于做回放差异验证。
  • 提前约定哪些字段属于动态字段，便于配置忽略规则，减少误报。
对方如果只能提供接口地址，而无法修改启动参数、无法挂 Agent、也无法提供日志或数据库核对渠道， 则可以做普通接口测试，但很难发挥本平台“真实录制回放”的核心价值。

六、已抓到的数据是否真实，可以从哪里验证

结论：可以验证为真实链路数据，而且当前仓库里已经有可核对的证据。

1

先看业务日志：入口请求真实到达了业务系统

`didi-system-a.log` 和 `didi-system-b.log` 中，能够看到真实请求进入入口接口 `/api/car/service`，并打印 `traId`、交易码、请求报文、响应报文、耗时。 例如 `traId=912359029940924300`、`traId=802217746546676000` 等都能在日志中直接定位。

/home/recording_playback_platform/runtime/logs/didi-system-a.log
/home/recording_playback_platform/runtime/logs/didi-system-b.log
2

再看子调用和数据库日志：同一笔交易的内部动作也真实发生

同一份 `didi` 日志里，能看到同一个 `traId` 对应的 HTTP 子调用请求 / 响应，以及数据库查询 / 更新。 这说明采集的不是单纯入口包，而是完整业务链路。

例如日志中可见： 数据库查询 `findVehicle / findPolicy / findClaim / findDispatch`， HTTP 子调用 `/internal/didi/risk`、`/internal/didi/pricing`， 以及数据库更新 `saveAudit`、`updateVehicleStatus`。

3

看平台原始表：Agent 上传的原始录制并未丢失

平台数据库中存在 `arex_mocker` 原始表，目前已有 1,023 条原始采集记录。 该表保留 `record_id`、`app_id`、类别、原始 JSON 数据，属于最靠近 Agent 上报的一层。

例如 `record_id = AREX-172-20-0-1-1833767948` 在原始表中可以同时看到 `Servlet`、`HttpClient`、`DynamicClass` 多种类别数据，说明同一笔交易的入口、子调用、数据库行为都已被采集。

平台表：arex_mocker
4

看同步后的录制表：原始数据已转成可用测试资产

平台数据库中 `recording` 表目前已有 95 条有效录制， 保留了请求方法、请求地址、请求体、响应体、交易码、子调用链路等关键字段。

仍以上述 `record_id = AREX-172-20-0-1-1833767948` 为例，`recording` 表里对应记录的交易码为 `car000015`，请求体和响应体中都能看到同一笔 `tra_id = 802217746546676000`，可直接与业务日志交叉核对。

平台表：recording / recording_session
5

看测试用例和回放结果：录制资产已能继续用于回归

平台中已生成 30 个测试用例，并已累计执行 21 次回放任务、 形成 75 条回放结果。这说明录制数据已经进入可复用、可回归的阶段。

平台表：test_case / replay_job / replay_result
6

最后看业务数据库：可核对业务副作用是否真实落库

对 `didi` 样例，日志中已经能看到 `saveAudit` 和 `updateVehicleStatus` 两类数据库写操作。 如果进一步登录业务库 `didi_alpha` / `didi_beta`，可以按 `request_no`、`txn_code`、`plate_no`、`quoted_amount` 等字段核对业务落库结果。

这意味着：不仅请求是真的，连处理后的数据库副作用也可以验证为真实发生。

换句话说，平台不是“自己拼了一份测试数据”。当前链路是：业务系统真实接收请求 -> Agent 采集并上报原始 mocker -> 平台原始表落库 -> 平台录制表整理 -> 生成测试用例和回放结果。 这条链路可以被逐层核验。
