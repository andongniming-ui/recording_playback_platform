## 一、P0 级别问题（高优先级，需立即处理）

### 1. 数据库迁移使用硬编码 ALTER TABLE，无版本控制
**位置**: `main.py` 第146-188行  
**现状**: 47条硬编码的 ALTER TABLE 语句，每次启动都尝试执行  
**问题**:
- 无法追踪数据库 schema 变化历史
- 没有回滚机制
- 迁移失败时难以调试
- 不支持多环境（dev/staging/prod）的版本一致性

**建议**: 使用 Alembic 进行数据库迁移管理

**优先级**: P0  
**预估工作量**: 4-6小时

---

### 2. secret_key 默认弱密钥无强制告警
**位置**: `config.py` 第14行  
**现状**: `secret_key: str = "changeme-in-production"`，虽有 `_validate_security_config()` 检查函数，但默认 `enforce_secure_secret=False`，仅打印警告而不阻止启动  
**问题**: 生产环境若忘记配置强密钥，会导致 JWT token 可被伪造，存在严重安全风险

**建议**:
- 生产环境（`debug=False`）强制要求修改 secret_key
- 或在检测到使用默认密钥时，日志中输出明显的红色告警

**优先级**: P0（安全）  
**预估工作量**: 0.5小时

## 二、P1 级别问题（中优先级，建议近期处理）

### 3. ArexClient HTTP 连接池未有效复用
**位置**: `api/v1/sessions.py` 第306行、`integration/arex_client.py`  
**现状**:
- `ArexClient` 类支持连接池复用（通过 `async with` 上下文管理器）
- 但 `sessions.py` 第306行直接实例化 `client = ArexClient(arex_url)`，未使用上下文管理器
- 导致每次调用 `health_check()` 等方法时可能创建新连接

**问题**: 高频调用时会产生大量短连接，影响性能

**建议**: 确保所有调用都通过 `async with ArexClient(...) as client:` 使用，或改进 `ArexClient` 自动管理连接生命周期

**优先级**: P1（性能）  
**预估工作量**: 2-3小时

---

### 4. 多进程部署时 ReplayJob 计数存在数据竞争
**位置**: `core/replay_executor.py` 第234行  
**现状**: 使用 `asyncio.Lock()` 保护 `done_count`、`passed`、`failed`、`errored` 等计数器  
**问题**: `asyncio.Lock()` 只能保护同一进程内的多协程。若使用 gunicorn 等多进程 worker 部署，此锁无效，会导致计数错误

**建议**:
- 若确认为单进程部署（如 uvicorn 单 worker），添加注释说明
- 若支持多进程，需使用 Redis 分布式锁或数据库原子操作（如 `func.coalesce(ReplayJob.passed, 0) + 1`）

**优先级**: P1（数据一致性）  
**预估工作量**: 1-2小时（调研）+ 2-4小时（修复）

---

### 5. HTML 报告生成代码内联在路由中（约250行）
**位置**: `api/v1/replays.py` 第779-1017行 `get_html_report()` 函数  
**现状**: HTML 模板字符串直接硬编码在 Python 函数中，包含大量 f-string 拼接  
**问题**:
- 代码难以维护和阅读
- 前后端职责混乱
- 容易出现 XSS 注入（需确认所有用户输入都正确转义）

**建议**: 使用 Jinja2 模板引擎，将 HTML 模板分离到独立模板文件

**优先级**: P2（代码质量）  
**预估工作量**: 3-4小时

**位置**: `api/v1/replays.py` `get_html_report()` 函数
