1. 一句话结论

录制回放平台之所以能录到 didi 项目的主请求、HTTP 子调用和数据库调用，不是因为平台直接去“监听”项目，而是因为项目启动时挂载了一个 Java Agent。这个 Agent 像一支“录音笔”，在项目运行过程中采集链路数据，并把采集结果回传给录制回放平台。

可以这样理解： 业务系统负责处理请求，Agent 负责录制，平台负责接收、整理、展示和回放。

2. 角色分工

业务项目

例如 didi-system-a、didi-system-b。负责真实处理业务请求，产生主流程、子调用和数据库操作。

AREX Agent

挂在 Java 进程里的采集组件。负责记录主请求、HTTP 子调用、数据库调用，并上报给平台。

录制回放平台

负责接收录制数据、管理录制会话、查看链路明细、生成测试资产并进行回放比对。

4. 平台为什么能录到请求

4.1 主请求

当外部调用业务接口时，例如 POST /api/car/service，请求先进入业务项目。由于 Agent 已经随着 Java 服务一起启动，它能在进程内部看到这次请求，因此可以记录请求体、响应体、状态码等信息。

4.2 子调用

复杂交易在处理过程中，往往会继续访问下游服务。在当前示例中，业务处理时会发起内部 HTTP 调用：

  • GET /internal/didi/risk
  • GET /internal/didi/pricing

这些虽然是系统内部自己调用自己，但本质上仍然是一次真实 HTTP 请求，因此 Agent 也能录到。

4.3 数据库调用

复杂交易还会访问 MySQL，例如查询车辆信息、查询客户信息、写入审计表、更新车辆状态。这些数据库动作通过 JDBC 执行，Agent 可以在运行过程中识别并采集。

5. 平台和项目如何关联

平台和项目真正关联起来，核心只有三件事：

关键条件 大白话解释
项目启动时挂载 Agent 相当于把“录音笔”装进业务项目里。没有这一步，平台录不到任何请求。
Agent 知道平台地址 相当于告诉录音笔“录完发到哪里”。地址或端口不对，平台就收不到数据。
平台和项目使用同一个应用标识 相当于给录制数据贴上项目姓名标签。这样平台才能把数据归到正确项目下。

6. 实际项目接入条件

  • 项目是 Java 服务，最好是 Spring Boot 类项目。
  • 项目启动方式可控，能够修改启动脚本、启动命令或容器启动参数。
  • 项目所在机器能访问录制回放平台的地址和端口。
  • 项目使用常见的数据库访问方式和 HTTP 调用方式，便于 Agent 识别。
  • 平台侧已创建应用，并配置好对应应用标识。

7. 实际项目接入步骤

  1. 先部署录制回放平台，并确认平台页面和后端可正常访问。
  2. 准备好 arex-agent.jar。
  3. 修改业务项目启动命令，挂载 -javaagent。
  4. 配置 Agent 上报目标地址和应用标识。
  5. 启动业务项目，确认 Agent 初始化成功。
  6. 在平台中创建对应应用。
  7. 在平台中开始录制。
  8. 发送真实业务请求。
  9. 停止录制并同步结果。
  10. 在平台中查看主请求、子调用、数据库调用和后续回放结果。

8. 常见失败原因

  • 没有挂载 -javaagent。
  • 平台地址或端口配置错误。
  • 平台没有先启动，导致 Agent 启动时无法建立录制链路。
  • 平台应用标识与项目应用标识不一致。
  • 网络、防火墙或容器网络导致平台与项目不通。
  • 业务请求虽然到了项目，但录制数据没有成功上报平台。

9. 管理价值

沉淀真实测试资产

把真实业务请求沉淀下来，形成后续可重复使用的测试样本，而不是只看一次性日志。

保留完整交易链路

不仅能看到入口请求，还能看到子调用和数据库行为，定位问题更快。

支撑发布回归验证

后续版本发布时，可以对同一批录制数据进行回放，快速判断是否出现回归问题。

管理层可以这样理解： 这套方案的核心价值，不是“多打了一些日志”，而是把真实交易沉淀成可重复执行、可持续复用、可量化对比的测试资产。

10. 总结

录制回放平台和业务项目的关联，本质上就是三步：

  1. 在业务项目里挂载 Agent。
  2. 让 Agent 把录制数据发回平台。
  3. 用统一的应用标识把平台和项目对应起来。

一旦接入成功，平台就能把“主请求 + 子调用 + 数据库调用”作为一条完整交易链路统一管理，并进一步用于录制沉淀、回放验证和问题排查。
