# didi 接口清单

本文档整理 `/home/recording_playback_platform/systems-under-test/didi` 下两个被测系统的全部 30 个逻辑接口，包含：

- 接口地址
- 请求方式 / 请求格式
- 交易码识别字段
- `tra_id` 链路追踪字段
- 子调用
- 数据库调用
- XML 请求示例

## 1. 系统与统一入口

两个系统对外入口完全一致，只是端口不同：

### 系统 A

- 系统名：`didi-system-a`
- AREX App ID：`didi-car-sat`
- 地址：`http://127.0.0.1:18081/api/car/service`

### 系统 B

- 系统名：`didi-system-b`
- AREX App ID：`didi-car-uat`
- 地址：`http://127.0.0.1:18082/api/car/service`

### 统一说明

- 请求方式：`POST`
- Content-Type：`application/xml` / `text/xml` / `text/plain`
- 请求体：XML
- 返回：XML
- 推荐每次请求都带 `<tra_id>`，用于串联主请求、HTTP 子调用、数据库调用日志
- 如果请求里不传 `<tra_id>`，系统会自动生成一个随机 18 位数字，并在响应 XML 中回传
- 交易码支持字段：
  - `code`
  - `trs_code`
  - `service_code`
  - `biz_code`
  - `trans_code`

### Postman 导入说明

- Collection 文件：`didi/postman/didi-car-systems.postman_collection.json`
- 环境文件：
  - `didi/postman/didi-system-a.postman_environment.json`
  - `didi/postman/didi-system-b.postman_environment.json`
- 导入后选择对应环境即可直接运行
- Collection 已内置 pre-request script，每次发送前会自动生成 18 位 `traId`
- 请求 XML 中会自动带上 `<tra_id>{{traId}}</tra_id>`

### 日志检索说明

- 主请求、HTTP 子调用、内部 `/internal/didi/*` 调用、数据库查询/更新日志都会带 `traId=...`
- 可直接这样查一笔完整交易：

```bash
grep 'traId=202604201823000001' logs/didi-system-a.log
```

## 2. 复杂接口统一调用链

以下接口属于“复杂接口”，会产生子调用和数据库访问：

- `car000001` ~ `car000015`

这些复杂接口的统一子调用链如下：

### HTTP 子调用

- `GET /internal/didi/risk`
- `GET /internal/didi/pricing`

说明：

- 子调用是系统内部通过 `RestTemplate` 发起，目标仍是本服务本机端口
- 录制时可以被 AREX 识别为 HTTP 子调用

### 数据库调用

读操作：

- `select * from car_vehicle where plate_no = ?`
- `select * from car_vehicle where vin = ?`
- `select * from car_customer where customer_no = ?`
- `select * from car_policy where policy_no = ?`
- `select * from car_policy where plate_no = ? order by policy_no limit 1`
- `select * from car_claim where claim_no = ?`
- `select * from car_claim where plate_no = ? order by claim_no limit 1`
- `select * from car_dispatch where garage_code = ? order by dispatch_no limit 1`
- `select * from car_dispatch where plate_no = ? order by dispatch_no limit 1`

写操作：

- `insert into car_order_audit(...) values (...)`
- `update car_vehicle set vehicle_status = ? where plate_no = ?`

### 相关业务表

- `car_customer`
- `car_vehicle`
- `car_policy`
- `car_claim`
- `car_dispatch`
- `car_order_audit`

## 4. 附加说明

### 4.1 复杂接口录制建议

如果你想在 `arex-recorder` 里看到更完整的录制链路，建议优先录以下交易码：

- `car000003`
- `car000006`
- `car000011`

原因：

- 既有 JDBC 子调用
- 也有 HTTP 子调用
- 在 A/B 两套系统之间还有可见 diff

### 4.2 简单接口用途

`car000016` ~ `car000030` 主要用于：

- 补齐 30 个逻辑接口数量
- 模拟同地址下的多交易码分流
- 作为无子调用 / 无数据库调用的对照样本
