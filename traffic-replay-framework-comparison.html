Sandbox、MoonBox、AREX 三套录制回放比对框架对比结论

结论先行：如果目标是 JDK8 + Spring Boot + 数据库与外呼录制 + 后续迁移到内网，且希望尽量少做二次开发， 公开资料更支持 AREX 优先、MoonBox 次选、Sandbox 体系更适合作为技术底座。其中 “Sandbox” 这里按行业常见说法理解为 Alibaba JVM-Sandbox / jvm-sandbox-repeater 体系；如果只看 jvm-sandbox 本体，它更像字节码增强容器，不是开箱即用的完整回放平台。

一页判断

无法通过公开资料证明“没有隐藏 bug”，因此本页把重点放在 成熟度、维护状态、平台完整度、许可证风险、数据库/外呼覆盖能力 上。 对金融场景更重要的，不只是能不能录制，还包括 能否在内网长期维护、许可证是否容易过审、依赖组件是否会带来合规和运维成本。

AREX
完整度最高。公开支持 HTTP/Dubbo/Redis/JDBC/MyBatis/Mongo/Kafka/RabbitMQ 等，覆盖数据库与外呼场景最直接。
MoonBox
业务平台化能力强，但 README 明确进入维护状态，且部署依赖 MySQL + Elasticsearch，落内网时需额外审查。
Sandbox
底层能力成熟，但 out-of-box 能力最弱，更适合技术团队做深度二开，而不是直接上平台。

三者详细拆解

Sandbox / Repeater

更准确地说，是 JVM-Sandbox + jvm-sandbox-repeater 组合。
定位：底座优先 二开工作量较大 许可证需重点审查
平台完整度58%
数据库/外呼覆盖清晰度68%
长期选型确定性61%
  • 优点：底层插桩能力成熟，Java Agent 模式对 Spring Boot 接入友好。
  • 公开插件：Java 入口、HTTP、Dubbo 入口；下游/数据侧可见 MyBatis、iBatis、Redis、Hibernate、Spring Data JPA 等插件。
  • 短板：项目更像“框架 + 插件”，不是带完整治理和报表的成品平台。
  • 适合：研发团队愿意自己补控制台、任务、审计、回放编排、报表、权限与内网运维。

MoonBox

vivo 基于 Repeater 演进的流量录制/回放平台。
平台化更强 维护态项目 Elasticsearch 需审查
平台完整度81%
数据库/外呼覆盖清晰度73%
长期选型确定性66%
  • 优点：有控制台、任务、机器管理、环境隔离等平台能力，落地形态接近可用产品。
  • 公开资料确认：支持录制下游 RPC、DB、缓存调用，并新增大量常用插件。
  • 公开使用线索：README 写明已在 vivo 运行 2 年，接入上百个项目。
  • 关键风险：README 明确写到 2025-04-30 起暂停迭代、进入维护状态，更适合作为参考实现或备选方案。

AREX

完整的录制回放与自动化 API 回归测试平台，公开生态最完整。
数据库/外呼覆盖最全 平台最完整 组件较多
平台完整度90%
数据库/外呼覆盖清晰度92%
长期选型确定性85%
  • 优点：中间件支持公开透明，平台组件齐全，对数据库写入校验和外呼回放场景最直接。
  • 公开支持：Servlet、Spring Web、Dubbo、HTTP Client、OkHttp、Redis、JDBC、MyBatis、MongoDB、Kafka、RabbitMQ 等。
  • 公开使用线索：官网写明在 Trip.com 内部服务于 4000+ 应用。
  • 关键约束：组件更多，迁入内网前需要把 MongoDB、Redis 与服务编排一次性规划好。

数据库、缓存、外呼与中间件支持清单

Sandbox / Repeater 已公开确认

  • 入口类：Java、HTTP、Dubbo。
  • 数据库/ORM：MyBatis、iBatis、Hibernate、Spring Data JPA。
  • 缓存：Redis。
  • 说明：HTTP 插件公开表述更偏“入口录制/回放”，对复杂外呼链路的公开透明度不如 AREX。

MoonBox 已公开确认

  • 基于 Repeater 深化，文档明确写到支持录制下游 RPC、DB、缓存调用。
  • 提供更多常用插件，但公开资料没有像 AREX 那样列出完整、逐项的插件清单。
  • 意味着它在能力上大概率够用，但正式采购/落地前最好做一次代码级插件盘点。

AREX 已公开确认

类别 支持项 与当前需求的关系
Web 入口 Servlet、Spring Web 适合 Spring Boot 被测服务录制请求入口
RPC / 外呼 Dubbo、Apache HttpClient、OkHttp 能直接覆盖常见 Java 外呼链路
数据库 JDBC、MyBatis、MongoDB 数据库读写录制与比对能力最直观
缓存 Redis 适合验证缓存依赖引起的回归差异
消息 Kafka、RabbitMQ 扩展到异步链路时更有优势

开源依赖与许可证关注点

框架 已公开可确认的核心依赖/组件 许可证与合规提示
Sandbox / Repeater JVM-Sandbox、jvm-sandbox-repeater、Alibaba TTL、Guava、Fastjson、OkHttp、SLF4J、Logback， 以及 MyBatis / iBatis / Redis / Hibernate / Spring Data JPA 等插件。 JVM-Sandbox 为 LGPL-3.0 / GPL-3.0；上层 Repeater 虽为 Apache-2.0，但金融内网正式使用前建议法务审查整个依赖链。
MoonBox MoonBox 自身 Apache-2.0；公开部署文档显示依赖 MySQL、Elasticsearch、Node.js。 从设计文档看其底层仍建立在 JVM-Sandbox Repeater 体系之上。 MoonBox 代码许可证友好，但 底层 JVM-Sandbox 与 Elasticsearch 的许可证都需要单独审查； MySQL Community 为 GPLv2。
AREX AREX Agent / API / Storage / UI 系列项目，公开架构依赖 MongoDB 与 Redis； Java 侧插件支持 Spring Web、Dubbo、HTTP Client、OkHttp、JDBC、MyBatis、MongoDB、Kafka、RabbitMQ 等。 AREX 代码 Apache-2.0；但 MongoDB Server 为 SSPL， Redis 从 7.4 起采用 RSALv2 / SSPLv1、Redis 8 起改为 AGPLv3 + SSPLv1 + RSALv2， 需要在企业开源合规清单中单列评估。
这部分是金融场景最容易被忽略的点：框架本身是开源，不等于 依赖的基础设施组件也都属于低风险许可证。 如果计划后续迁到内网，建议在 PoC 前就把 JVM-Sandbox、MongoDB、Redis、Elasticsearch、MySQL 的许可证与版本策略一起过一遍。

不容易第一眼想到，但实际很关键的点

许可证不是小问题

高优先级

Sandbox 体系底层的 JVM-Sandbox 不是 Apache/MIT 这类最宽松许可证；AREX 与 MoonBox 虽然代码本身更友好，但 MongoDB、Redis、Elasticsearch 都需要单独看版本与许可。

平台完整度直接影响内网迁移成本

中高优先级

仅有插桩能力不够，真正进内网后还会碰到任务调度、数据保留、权限、报表、对比基线、失败审计与运维接管。AREX、MoonBox 在这方面明显强于 Repeater。

“支持中间件”与“支持得足够透明”是两回事

中优先级

AREX 的插件清单更公开，适合做正式选型。MoonBox 能力未必弱，但公开插件 inventory 不如 AREX 透明，正式落地前最好做代码级盘点。

如果希望把风险收敛到最小，建议后续 PoC 就按下面顺序做：1. 先用 AREX 验证 JDBC / MyBatis / HTTP / Dubbo 主链路；2. 同时审查 MongoDB / Redis 许可策略；3. 再决定是否需要拿 MoonBox 或 Repeater 作为备选或底座方案。

最终建议

场景 建议
希望尽快做出可演示、可推广的平台 优先评估 AREX。它对数据库、缓存、消息和外呼支持更清晰，最接近“装上就能做事”的形态。
希望参考国内团队的平台化实践 MoonBox 可以作为第二选择，尤其适合借鉴控制台和平台化设计，但不建议把它当成唯一长期路线。
希望打造完全自主可控的底座 可研究 Sandbox / Repeater，但它更像“自己做平台的内核”，不适合直接当最终产品。
以当前需求做单一推荐 AREX 更匹配。原因不是它一定“绝对没有 bug”，而是公开资料下它的 插件覆盖、平台完整度、数据库/外呼适配、使用规模线索 都更强。
