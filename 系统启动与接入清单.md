## 5. 平台“新增应用”要填写什么

平台页面：`平台 -> 应用管理 -> 新增应用`

### 5.1 字段说明表

| 字段 | 是否必填 | 说明 | 当前建议 |
|---|---|---|---|
| 应用名称 | 是 | 平台内显示名称 | 建议和服务目录或环境名一致，例如 `didi-system-a` |
| 描述 | 否 | 补充说明 | 可写环境、用途、是否基线 |
| 宿主机地址 | 是 | 被测系统所在机器的 IP 或域名 | 本机联调可填 `127.0.0.1`，远端部署填真实 IP |
| 宿主机用户 | 是 | SSH 登录用户 | 当前本机常用为 `test` |
| 宿主机端口 | 否 | SSH 端口 | 默认 `22` |
| 启动模式 | 是 | `ssh_script` 或 `docker_compose` | 当前这些样例系统建议 `ssh_script` |
| Docker 工作目录 | Docker 模式必填 | 远端 Docker Compose 所在目录 | 只有 `docker_compose` 模式才填 |
| Docker Compose 文件 | Docker 模式可选 | Compose 文件名 | 默认 `docker-compose.yml` |
| Compose 服务名 | Docker 模式必填 | `docker compose up` 的 service 名 | 只有 `docker_compose` 模式才填 |
| Docker Agent Storage | 否 | Docker 模式下 agent 使用的 storage 地址 | 留空一般走平台默认 |
| Agent 挂载路径 | 否 | Docker 模式下容器内 agent jar 路径 | 默认 `/opt/arex/arex-agent.jar` |
| 密钥路径 | 否 | SSH 私钥路径 | 例如 `/home/test/.ssh/id_rsa` |
| 连接密码 | 否 | SSH 密码 | 用密钥时可不填 |
| 服务端口 | 否但强烈建议填 | 被测服务实际监听端口 | 必须和服务真实端口一致 |
| JVM 进程名 | 否但强烈建议填 | 平台通过 `pgrep` 识别 Java 进程 | 一般填 jar 文件名 |
| AREX App ID | 否但录制同步强烈建议填 | 必须和 agent 上报的 `service.name` 一致 | didi 必填，其他系统接 AREX 后再对齐 |
| AREX Storage 地址 | 否 | 单应用覆盖 storage 地址 | 留空则走全局配置 |
| 采样率 | 否 | 录制采样率 | 默认 `1.0` 即 100% |
| 交易码映射 | 否 | 回放前按交易码改写请求/响应字段 | 有特殊报文映射需求时再填 |

### 5.2 填写时的关键原则

| 项 | 原则 |
|---|---|
| 宿主机信息 | 填“服务实际运行在哪台机器，就写哪台机器” |
| 服务端口 | 填业务服务端口，不是 SSH 端口 |
| JVM 进程名 | 填 `pgrep -f` 能匹配到的 jar 名或唯一进程关键字 |
| AREX App ID | 必须和 `-Darex.service.name` 或 Agent 实际上报的 `appId` 完全一致 |
| 启动模式 | 当前样例系统基本都适合 `ssh_script`，只有远端 Compose 部署才用 `docker_compose` |

## 7. 当前最实用的接入建议

| 场景 | 建议 |
|---|---|
| 需要马上验证录制回放链路 | 优先接 `didi-system-a` 和 `didi-system-b`，因为这两套已经挂了 AREX Agent |
| 需要只看业务接口差异 | `waimai` 和 `loan-system` 可以先独立启动、直接调接口 |
| 需要把 `waimai` / `loan-system` 也纳入平台录制 | 后续给它们的 `start.sh` 补 `-javaagent`，再设置稳定的 `AREX App ID` |
